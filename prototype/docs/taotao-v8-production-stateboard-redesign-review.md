# 桃桃 V8 生产级状态板重构评审稿

> 本文件用于产品评审和下一轮原型重构，不进入用户端界面。  
> 当前结论：现有 V7.8 已经有很多正确状态，但组织方式仍像“状态合集”，不是能直接指导开发的“功能链路状态板”。

## 1. 本轮目标

把聊天原型从“散点手机稿”重构为“按功能链路展开的一排手机稿”。

目标不是补几个状态，而是让研发、设计、产品能顺着每个功能看到：

1. 用户从哪里触发。
2. 当前手机界面是什么。
3. 下一步是什么。
4. 分支怎么走。
5. 异常怎么回收。
6. 小雨视角和阿川视角有什么不同。
7. 哪些内容只在文档里解释，不能出现在手机 UI 里。

最终状态板应像 Figma 评审稿：每条功能一条横向泳道，每一步都是一个真实手机稿，分支和异常展开在同一条链路下，而不是混在一个大网格里。

## 2. 产品原则重新锁定

### 2.1 聊天是主产品

桃桃不是任务管理器，也不是审批系统。默认界面是小雨、阿川、桃桃三个人聊天。

普通聊天、情绪接话、玩梗、媒体消息、已读、撤回、引用、失败，都必须留在聊天语境里，不进入事务系统。

### 2.2 桃桃是第三个人

桃桃可以说话，但不能像系统广播器。

允许：

- 接梗。
- 陪聊。
- 递话。
- 帮用户整理一句话。
- 在明确请求时生成结果。

不允许：

- 主动播报“对方看过了”。
- 主动播报“发送失败”。
- 反复说“我放这儿”。
- 用产品状态替代人话。

### 2.3 顶部胶囊是索引，不是警报栏

顶部胶囊常驻，只做“今天的小事”的索引和找回。

点击顶部胶囊后，先打开今天的小事抽屉，再进入详情或回到原消息。

发送失败不进入顶部胶囊，只留在原气泡下。

### 2.4 消息内胶囊从桃桃消息长出来

群聊里用户明确让桃桃做事时，正确顺序是：

1. 用户说话。
2. 桃桃像第三个人回一句。
3. 桃桃这条气泡下出现小胶囊。
4. 胶囊显示生成中、已完成、失败或可重试。
5. 用户点击胶囊，才打开详情半窗。

不能在用户刚说完一句话后无逻辑弹半屏。

### 2.5 小事卡是通用对象

创建一件事就是创建一张结构化小事卡：

- 标题
- 时间
- 地点
- 备注
- 参与人
- 是否需要对方确认

AI 负责帮用户补齐、改写、生成，不替代对象本体。

保存、发送、请对方确认，是同一张卡的三个去向。

### 2.6 保存是私有去向

用户点保存后：

1. 半窗收起。
2. 群聊不出现新气泡。
3. 桃桃不发保存回执。
4. 只有本人顶部胶囊出现私有索引。
5. 点胶囊后再进入详情。

保存不是发到群聊，不能污染两个人的聊天。

### 2.7 发送失败按聊天消息处理

所有发送失败，包括普通消息、卡片发送、请对方确认、改动发回，都参考微信：

- 停在原消息位置。
- 气泡下显示轻提示。
- 点原消息或失败提示重试。
- 不弹半屏。
- 不进顶部胶囊。
- 不由桃桃广播。
- 对方不可见。

## 3. 当前主要问题

### 3.1 状态板组织方式不对

当前 `ChatInterfaceBoard` 直接把 `chatDisplayStates` 扁平渲染到一个大网格里。

问题：

- 手机稿按全局序号排列，不按功能链路排列。
- 同一功能的开始、分支、异常、另一方视角散在不同位置。
- 上方的流程说明和下方手机稿是分离的，研发读完流程后还要自己找状态。
- 响应式网格会自动换行，同一链路在不同屏宽下视觉顺序会漂移。

结论：现有结构适合“展示已有状态数量”，不适合“指导生产开发”。

### 3.2 flow.steps 覆盖不完整

已有 `chatProductionFlows` 是正确方向，但很多 state 只挂在 `flow.stateIds`，没有进入 `flow.steps`。

典型问题：

- `private_create` 缺草稿保存、退出未完成、草稿列表、恢复草稿、另一方不可见视角。
- `petition_review` 把大量状态压进少数步骤，漏掉发起方追踪、接收方晚点、驳回、失败、二次往返。
- `exceptions` 漏重复发送、长期未读。
- `memory_reminder` 把记忆和提醒权限塞在一起，但没有成功结束态。

结论：现在更像目录，不是可执行链路图。

### 3.3 有些状态归属不顺

需要重点修正：

- `s3k_partner_view_no_private_draft` 是私下创建的另一方视角，不应被 `quiet_presence` 自动归到群聊能力。
- `s4c_card_read_collapsed` 是对方已看后的过渡状态，更接近请对方确认链或顶部索引，不应只散在目的地 flow。
- 提醒权限失败和异常边界有交叉，但提醒功能本身缺少成功链路。

### 3.4 动作跳转不能只靠按钮文案推断

当前 `actionIdByLabel` 和 `nextStateByActionId` 是全局映射，容易导致同一按钮在不同链路里跳错。

例如：

- `发送` 在不同状态下可能是发送到群聊、重新发送、发送修改版，不应该都跳一个状态。
- `生成预览` 在私下创建和修改上书里不是同一条链路。
- `发送` 在接收方修改和发起方二次修改里需要 state 级动作覆盖，不能靠全局按钮文案推断。

下一轮应支持 state 级动作覆盖。

### 3.5 用户端文案仍有内部语言

手机 UI 中仍有一些会破坏 C 端体验的过程词。

明确保留：

- 上书
- 准奏
- 驳回

这三个词作为桃桃的产品仪式语言保留在用户端；它们不是后台审批词，而是情侣之间轻量、带一点玩笑感的处理口径。

需要从手机 UI 移除：

- 当前
- 上一版
- 版本
- 发回去
- 回传
- 原消息
- 小事卡
- 生成预览
- 未生成
- 我放这儿

这些词可以出现在产品文档里，但不应该出现在用户手机界面里。

推荐口径：

| 内部词 | 用户端口径 |
| --- | --- |
| 当前 | 这次想法 |
| 上一版 / 版本 | 改动时间线：小雨发起、阿川改动、20:14 写下 |
| 发回去 / 回传 | 发送 |
| 原消息 | 回聊天 |
| 小事卡 | 这件事 / 小事 |
| 生成预览 | 整理一下 / 整理好了 |
| 未生成 | 还没整理好 |

改动历史采用飞书文档式时间线：详情默认展示最新内容，下面按时间显示谁在什么时候发起、改动、准奏或驳回。用户能点开历史条目看发生了什么，但不出现“上一版 / 版本 / 回传 / 原消息”这类过程标签。

## 4. 新状态板信息架构

### 4.1 总体结构

新增一个链路化状态板，而不是继续使用扁平网格。

结构：

```text
V8 状态板总览
  Flow Lane 1: 进入聊天前
    Step Strip: 登录
      手机稿...
    Step Strip: 创建桃桃
      手机稿...
    Step Strip: 绑定伴侣
      手机稿...

  Flow Lane 2: 三人聊天底座
    Step Strip: 正常聊天
    Step Strip: 消息状态
    Step Strip: 轻入口
    Step Strip: 媒体和长按

  Flow Lane 3...
```

每个 Flow Lane 是一个功能，每个 Step Strip 是一排手机稿。

### 4.2 每张手机稿必须带的信息

手机界面只显示用户真实可见内容。

手机稿下方可以显示评审信息：

- 当前功能链路
- 当前步骤
- 用户动作
- 系统状态
- 下一步
- 异常回收
- 小雨视角 / 阿川视角
- 研发字段

但这些评审信息不能进入手机屏幕内部。

### 4.3 推荐 data-testid

下一轮实现时新增：

- `v8-chatboard-journeys`
- `chatboard-flow-lane`
- `chatboard-flow-header`
- `chatboard-flow-step`
- `chatboard-step-state-strip`
- `chatboard-step-state-slot`
- `chatboard-flow-fallback-step`
- `chatboard-unmapped-state`

验收要求：

1. 所有 state 必须出现一次。
2. 每个 state 必须属于一个 flow。
3. 每个 flow 的核心 state 必须进入 step，不允许只挂在 `flow.stateIds`。
4. 允许有 fallback，但 fallback 必须列明原因。
5. 手机 UI 内不得出现内部说明和状态机语言。

## 5. 应重排的功能链路

### 5.1 Flow A：进入聊天前

目标：确保先登录，再创建桃桃/绑定伴侣，不能未登录先聊。

应展开手机稿：

1. App 登录入口：手机号登录。
2. 小程序入口：微信授权。
3. 验证码输入。
4. 验证码失败。
5. 未绑定伴侣。
6. 上传照片创建桃桃。
7. 不上传照片，使用默认桃桃。
8. 生成桃桃中。
9. 生成失败。
10. 生成成功。
11. 发邀请。
12. 邀请已发出。
13. 伴侣打开邀请。
14. 伴侣确认加入。
15. 伴侣拒绝或链接失效。
16. 双方进入三人聊天。

当前缺口：

- 缺验证码失败细分。
- 缺邀请已发出和伴侣接受链路。
- 创建桃桃成功后的落点不清晰。

### 5.2 Flow B：三人聊天底座

目标：证明这是聊天 App，不是任务 App。

应展开手机稿：

1. 空聊天。
2. 默认聊天。
3. 叫桃桃。
4. 输入框聚焦。
5. 消息发送中。
6. 消息失败可重试。
7. 撤回。
8. 引用回复。
9. 已送达未读。
10. 对方已读。
11. 对方正在输入。
12. 图片。
13. 语音。
14. 表情。
15. 长按自己消息。
16. 长按桃桃消息。
17. 加号面板。

当前缺口：

- 这些状态已有，但没有按聊天底座完整排列。
- 长按和加号入口需要明确“轻入口”，不能变成任务面板。

### 5.3 Flow C：桃桃介入层级

目标：定义桃桃什么时候不说、轻说、接梗、陪聊、递话、生成。

应展开手机稿：

1. 沉默在场。
2. 轻反应。
3. 接梗。
4. 情绪陪伴。
5. 模糊意图不升级。
6. 明确 @ 桃桃。
7. 明确请求能力。
8. 安全边界。

当前缺口：

- 这些状态目前混在 `group_capability`，不像一条介入策略链。
- 桃桃文案还偏系统播报，需要重写成第三个人说话。

### 5.4 Flow D：群聊里长出 AI 能力

目标：AI 能力从聊天里自然长出来，不打断聊天。

应展开手机稿：

1. 用户请求：“给个做法。”
2. 桃桃先回一句。
3. 桃桃气泡下生成中胶囊。
4. 生成进度。
5. 生成完成。
6. 用户点击胶囊。
7. 结果详情半窗。
8. 用户输入修改方向。
9. 原胶囊更新中。
10. 修改完成。
11. 修改失败，旧版保留。
12. 用户收起详情。
13. 回到原消息胶囊。
14. 用户把结果保存成小事。

当前缺口：

- 缺首次生成失败分支。
- 缺“保存 AI 结果为小事”的完整去向。
- 顶部胶囊和消息内胶囊的边界需要继续强化。

### 5.5 Flow E：私下找桃桃创建小事

目标：用户主动创建一件事，AI 帮忙补齐字段。

应展开手机稿：

1. 从输入区 + 进入。
2. 从顶部胶囊 + 进入。
3. 私下半窗空态。
4. 用户输入需求。
5. 桃桃追问缺失字段。
6. 用户补字段。
7. 生成中。
8. 生成失败。
9. 生成成功，显示小事预览。
10. 用户修改字段。
11. 用户退出未完成。
12. 保存草稿。
13. 草稿列表。
14. 恢复草稿。
15. 另一方视角：完全不可见。

当前缺口：

- 草稿生命周期没有完整排进 steps。
- 另一方不可见视角归属错误。
- 创建失败和发送失败边界需要继续区分。

### 5.6 Flow F：保存 / 发送 / 请对方确认

目标：同一张预览卡的三个去向要清楚。

#### 保存

1. 预览页点保存。
2. 半窗收起。
3. 群聊不新增气泡。
4. 本人顶部胶囊出现私有索引。
5. 点胶囊打开详情。
6. 详情里可修改、发送、请对方确认、删除。

#### 发送

1. 预览页点发送。
2. 发送前确认。
3. 发送中。
4. 发送失败，原气泡重试。
5. 发送成功，群聊出现轻入口。
6. 对方点开。
7. 入口变轻。

#### 请对方确认

1. 预览页点请对方确认。
2. 发送中。
3. 发送失败，原气泡重试。
4. 发送成功，群聊出现重点入口。
5. 对方顶部胶囊出现“等你看看”。

当前缺口：

- 保存刚刚已纠正，但状态板还需要在链路里明确它不是聊天消息。
- `s4c_card_read_collapsed` 需要放进对方打开后的链路。
- “上书 / 准奏 / 驳回”保留为用户端口径，但周边文案不能出现版本、发回、回传等过程词。

### 5.7 Flow G：请对方确认 / 双方往返

目标：这是目前最需要重做的链路。

应拆成六段。

#### G1 发起方发出后

1. 发起方发出。
2. 对方未打开。
3. 发起方可撤回。
4. 对方已打开但未决定。
5. 对方选择晚点。
6. 对方拒绝。

#### G2 接收方第一次打开

1. 接收方在聊天入口看到。
2. 接收方从顶部胶囊看到。
3. 打开详情。
4. 选择同意。
5. 选择修改。
6. 选择先不这样。

#### G3 接收方修改

1. 进入私下修改。
2. 和桃桃说修改方向。
3. 生成中。
4. 生成失败。
5. 修改预览。
6. 发给对方看。
7. 发送失败。
8. 成功回到群聊。

#### G4 发起方看修改

1. 发起方收到修改入口。
2. 从顶部胶囊看到待处理。
3. 打开详情。
4. 同意。
5. 再改。
6. 先不这样。

#### G5 二次往返

1. 发起方再改。
2. 生成中。
3. 生成失败。
4. 预览。
5. 发给对方看。
6. 对方再次打开。
7. 最终同意或收起。

#### G6 结束

1. 双方同意。
2. 顶部胶囊降级。
3. 详情可回看。
4. 一段时间后进入最近小事。

当前缺口：

- 现有 `petition_review.steps` 压得太粗。
- 发起方追踪和接收方视角没有分开。
- 多次修改的状态关系不够清晰。
- 用户端仍出现“版本 / 发回 / 回传”等状态机词；上书、准奏、驳回保留。

### 5.8 Flow H：顶部胶囊 / 今天的小事

目标：顶部胶囊提供掌控感和找回，不干扰聊天。

应展开手机稿：

1. 空态。
2. 单个待处理。
3. 多个小事。
4. 私有草稿。
5. 私有保存。
6. 对方已看未定。
7. 对方改了等我看。
8. 已定。
9. 打开今天的小事抽屉。
10. 点条目进入详情。
11. 点条目回到刚才那句。
12. 删除草稿。
13. 收起抽屉。

当前缺口：

- 抽屉已有，但优先级冲突规则需要开发化。
- 文案里的“原消息/来回记录”需要用户化。
- 旧文档还写发送失败进入顶部胶囊，必须改掉。

### 5.9 Flow I：记忆

目标：记忆是完成后的轻沉淀，不是后台能力展示。

应展开手机稿：

1. 小事完成后出现轻入口。
2. 桃桃提议留一条记忆。
3. 双方可看。
4. 一方修改。
5. 一方选择不留。
6. 双方确认后进入小窝。
7. 之后可回看。

当前缺口：

- 缺最终沉淀态。
- 用户端不要解释“记忆系统”。

### 5.10 Flow J：提醒

目标：提醒是小事的辅助能力，不抢主流程。

应展开手机稿：

1. 小事里设置提醒。
2. 个人提醒。
3. 双方提醒。
4. 权限未开。
5. 打开权限引导。
6. 权限失败。
7. 权限成功。
8. 提醒已设置。
9. 取消提醒。

当前缺口：

- 现在只有权限失败，没有完整成功链路。

### 5.11 Flow K：异常和边界

目标：所有异常都自然回到聊天。

应展开手机稿：

1. 发送失败。
2. 生成失败。
3. 重复发送。
4. 长时间未读。
5. 桃桃理解错。
6. 用户纠偏。
7. 误触发创建。
8. 用户取消误触发。
9. 安全边界。
10. 网络恢复。

当前缺口：

- 重复发送、长期未读没有进 steps。
- 安全边界文案仍有内部策略外露。

## 6. 用户端文案红线

以下词禁止出现在手机 UI 中：

- 状态机语言：当前、上一版、版本、回传、待发回、原消息。
- 保留口径：上书、准奏、驳回。
- 过度审批语言：批阅、等你批。
- 内部对象语言：小事卡、生成预览、未生成、生命周期、状态。
- 系统广播语言：我放这儿、已保存、已上书、对方已看、发送失败。
- 解释性文案：还没出现在聊天里、对方看不到、不会打断聊天、先留着以后再改。

允许在评审区出现，但手机屏幕内部不能出现。

## 7. 下一轮实施顺序

不要一次性重写所有 UI。建议分三步。

### Step 1：先做新状态板外壳

只改展示组织方式：

- 新增 Flow Lane。
- 新增 Step Strip。
- 复用现有 ChatStateCard。
- 所有状态只出现一次。
- 保留 fallback 区。

完成后，状态板先变得能读。

### Step 2：补齐 flow.steps

按本文件把每条功能链路补完整。

优先级：

1. 请对方确认 / 双方往返。
2. 私下创建 / 草稿生命周期。
3. 保存 / 发送 / 请对方确认三去向。
4. 顶部胶囊索引。
5. 记忆和提醒。
6. 异常。

### Step 3：清理用户端文案

手机界面只保留用户能看懂的表达。

内部解释移到评审区或文档。

## 8. 验收标准

### 8.1 状态板验收

- 每条功能是一条 lane。
- 每个 step 是一排手机稿。
- 主路径、分支、异常、双方视角都展开。
- 研发能不看源码理解下一步。
- 所有 state 只出现一次。
- fallback 状态数量必须可解释，最终趋近 0。

### 8.2 手机 UI 验收

- 手机屏幕内部没有内部术语。
- 桃桃像第三个人，不像系统通知。
- 普通聊天不被事务化。
- 发送失败只在原气泡。
- 保存不进群聊。
- 顶部胶囊只做索引。
- 半屏只来自用户主动点击。

### 8.3 研发交付验收

- 每个 state 有唯一 id。
- 每个 action 有当前 state 下的明确 nextState。
- 每个结构化对象有状态字段。
- 每个异常有回收路径。
- 每个私有状态有另一方不可见证明。
