# 桃桃后续功能分册索引

更新日期：2026-06-24

本目录用于承接 V7.8 聊天状态板和 V7.9 桃桃页分册之后的功能规划。每个功能单独成册，后续开发时只取当前模块、当前 MVP 基线和必要状态板，不把所有历史 PRD 塞进同一个上下文。

## 当前冻结基线

当前最小 MVP 已基本覆盖：

- 手机号登录与验证码。
- 登录后未绑定也能使用。
- 邀请另一半：微信分享和手机号搜索两条路径。
- 三人聊天主界面：小雨、阿川、桃桃。
- 顶部胶囊：邀请、今天的小事、待处理和找回。
- 聊天里的小事生成、修改、保存、发送、上书、准奏、驳回。
- 桃桃页：从聊天进入、身份展示、轻量改名、日历回看。

后续功能不得反向改写这套 MVP 基线，除非先在当前分册里明确说明影响范围和迁移方案。

## 使用顺序

1. 先读 `docs/product/14-v7.8-production-chatboard-current.md`，确认当前聊天、登录、邀请和小事卡规则。
2. 再读 `docs/backlog/01-taotao-life-page.md`，确认桃桃页已冻结的入口、改名和日历规则。
3. 再读本索引，选择本次要开发的功能分册。
4. 修改原型时，只把对应分册转成状态板、截图和验收脚本。
5. 功能上线前，再把实际实现回填到对应分册，不反向污染当前 MVP 文档。

## 分册约束

- 一个功能一个文档，不合并成全量 PRD。
- 每次只推进一个功能域，避免开发上下文过大。
- 当前 MVP、后续增强、商业化设想必须分开写。
- 原型手机稿只展示用户会看到的内容；开发说明留在分册或状态板旁注。

1. `03-reminder-notification.md`：再做提醒通知，保证小事不会丢。

4. `05-feeding-gift.md`：投喂礼物等商业化触发先后置，避免污染早期体验。

## 已合并不单独开册

- 小窝记忆不再作为独立功能模块推进。小事沉淀、回看和轻记忆能力并入 `01-taotao-life-page.md` 的桃桃页日历体系；后续只在桃桃页和聊天小事详情里补状态，不再做单独小窝 Tab 或小窝首页。

## 写作规则

- 用户端文案只写用户真的会看到的字。
- 技术字段、状态机和验收点写在文档表格里，不进手机界面。
- 新功能必须说明入口、可见性、状态、异常和验收。
- 没有明确用户价值的功能不进入原型。
