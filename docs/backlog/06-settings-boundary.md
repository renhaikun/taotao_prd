# App 基础控制分册

更新日期：2026-06-24

## 分册定位

当前 V7.8 聊天主界面和 V7.9 桃桃页已经覆盖最小 MVP：登录、邀请、三人聊天、小事生成与处理、桃桃身份和日历回看。

本分册补齐早期 App 必须存在的基础控制能力，但必须保持简单纯粹。桃桃是 AI-first 的关系陪伴 App，不需要把传统设置页里的功能全部搬出来。

当前阶段只解决四件事：

- 我是谁：账号和登录状态。
- 谁和我绑定：另一半关系状态。
- 哪里不好用：早期反馈入口。
- 低频控制：通知、安全、关于、删除账号等放进更多。

## 这次收敛后的判断

- `桃桃记住的内容` 不作为设置入口曝光。它属于桃桃页、聊天和日历体系里的能力，不需要在设置里单独解释。
- `可见范围` 当前不是 MVP 必须项，不在首页展示，也不单独做页面。
- 未来如果做发现/类似朋友圈，权限控制应该在用户执行分享动作时出现，而不是现在提前放一个设置项。
- 解除关系必须保留，但放在 `另一半` 二级页里，不放首页。
- 反馈是早期 App 必须有的入口，需要首页可见。

## 设计原则

- 不做底部 Tab。
- 不做“小窝”独立页面；小事沉淀和回看已经并入桃桃页日历体系。
- 不做传统功能菜单。
- 不做三人关系大卡。
- 不在首页放可见范围、数据边界、桃桃记住内容。
- 不在用户界面出现技术词、状态机词或内部说明。
- 从聊天自然进入，也能安全返回聊天。

## 推荐入口

| 入口 | 说明 | MVP 是否需要 |
| --- | --- | --- |
| 聊天页当前用户头像 | 用户点自己的头像进入我的 | 做 |
| 未绑定状态的邀请胶囊 | 仍只进入邀请流程，不进入我的 | 已在 V7.8 |
| 桃桃页左上返回 | 只返回聊天，不进入我的 | 已在 V7.9 |

入口命名建议用 `我的`，比 `设置` 更轻，也更适合早期 AI-first App。

## 首页结构

首页只保留三个直接入口和一个低频入口：

| 区块 | 用户理解 | 点击后 |
| --- | --- | --- |
| 账号 | 当前登录的是我 | 账号详情 |
| 另一半 | 当前是否已绑定 | 关系详情 |
| 反馈 | 哪里不好用，告诉我们 | 反馈页 |
| 更多 | 通知、安全、关于 | 更多设置 |

首页不直接放：

- 可见范围。
- 桃桃记住的内容。
- 微信邀请和手机号绑定细节。
- 三人关系大卡。
- 解除关系。
- 删除账号。
- 拉黑、举报等强动作。
- 大段隐私说明。

## 账号详情

账号页只做基础身份和登录控制。

| 能力 | MVP 是否做 | 说明 |
| --- | --- | --- |
| 查看手机号 | 做 | 只展示脱敏手机号 |
| 微信绑定状态 | 做 | 只展示状态，不展开复杂绑定流程 |
| 退出登录 | 做 | 二次确认 |
| 删除账号 | 做入口 | 进入删除账号说明和二次确认 |
| 换绑手机号 | 后置 | 需要风控，先不进 MVP |

用户端文案建议：

- `手机号`
- `微信`
- `退出登录`
- `删除账号`

## 另一半

关系页只保留必要信息，不做复杂三人关系展示。

### 已绑定

| 内容 | UI |
| --- | --- |
| 另一半 | 昵称和头像 |
| 绑定状态 | `已绑定` |
| 解除关系 | 二级危险操作 |

不展示三人关系大图，不展示桃桃在中间的大关系卡。

### 未绑定

| 内容 | UI |
| --- | --- |
| 当前状态 | `还没有绑定另一半` |
| 邀请入口 | `邀请另一半` |

邀请仍然回到 V7.8 已定义的邀请半屏，不在关系页重新设计一套邀请流程。

### 解除关系

解除关系必须说明：

- 你们将不再是绑定关系。
- 聊天不会继续同步。
- 小事和日历如何处理。
- 桃桃名字和你的账号仍保留。
- 如需重新绑定，需要重新邀请。

解除关系必须二次确认，不和删除账号复用同一个文案。

## 反馈

反馈是早期 App 必须有的入口。它不是客服中心，也不是复杂工单系统，而是帮助我们尽快发现真实用户不顺手的地方。

### 首页反馈入口

首页保留一个轻入口：

- 文案：`反馈`
- 副文案：`哪里不好用，直接告诉我们`
- 位置：在账号、另一半之后，更多之前
- 视觉：轻行项目，不做大卡，不做强运营入口

### 反馈页

| 区块 | 用户看到 | 操作 |
| --- | --- | --- |
| 问题描述 | 输入遇到的问题或建议 | 必填 |
| 联系方式 | 手机号默认带入，可修改 | 可选 |
| 相关截图 | 后续可添加 | MVP 可后置 |
| 提交 | 发送反馈 | 成功或失败 |

用户端文案建议：

- `哪里不好用，直接告诉我们`
- `问题或建议`
- `提交反馈`
- `已收到`

避免：

- `工单`
- `ticket`
- `客服系统`
- `问题类型编码`
- `日志上传`

## 更多

低频项进入 `更多`，不抢首页第一视觉：

| 能力 | 是否做 | 放置 |
| --- | --- | --- |
| 通知 | 做轻版 | 更多 |
| 安全 | 做 | 更多 |
| 关于桃桃 | 做 | 更多 |
| 删除账号 | 做 | 账号页和更多均可到达 |

提醒能力后续由 `03-reminder-notification.md` 单独展开。

## 关于可见范围

当前不做独立可见范围入口。

原因：

- 当前没有发现/朋友圈式公开场景。
- 提前给用户一个可见范围设置，会让产品显得复杂。
- 早期 MVP 的默认心智应该是：聊天、小事和日历都在你们之间。

未来如果上线发现或分享能力，可见范围应该在具体动作里出现：

- 用户点击分享某件小事。
- 系统弹出分享预览。
- 用户选择 `仅我们可见`、`发给朋友` 或 `发布到发现`。
- 用户确认后才公开。

因此，可见范围作为未来能力保留在产品架构里，但不进入当前基础控制原型。

## 关于桃桃记住的内容

当前不做独立入口。

原因：

- 桃桃记住什么，本来就在桃桃页、聊天和日历里体现。
- 单独做一个“桃桃记住的内容”会让用户感觉我们在解释系统，而不是让产品自然工作。
- 如果用户需要改名或回看小事，去桃桃页。
- 如果用户需要删除账号或解除关系，去账号或另一半。

后续如需更透明的数据说明，可以放在 `更多 > 关于桃桃` 或隐私政策中，不作为首页功能。

## 页面状态清单

后续做高保真状态板时，至少展开以下手机稿：

| 状态 ID | 页面 | 用户动作 | 下一步 |
| --- | --- | --- | --- |
| mine_entry_chat | 聊天页入口 | 点当前用户头像 | 打开我的 |
| mine_home_bound | 我的，已绑定 | 点账号/另一半/反馈/更多 | 进入对应页 |
| mine_home_unbound | 我的，未绑定 | 点邀请另一半 | 回到邀请半屏 |
| account_detail | 账号详情 | 查看手机号/微信/退出 | 二次确认或返回 |
| logout_confirm | 退出登录确认 | 确认退出 | 回登录页 |
| relationship_bound | 另一半，已绑定 | 查看另一半/解除关系 | 进入解除说明 |
| relationship_unbound | 另一半，未绑定 | 邀请另一半 | 回邀请半屏 |
| relationship_unbind_review | 解除关系说明 | 继续 | 二次确认 |
| relationship_unbind_confirm | 解除关系确认 | 确认解除 | 回未绑定状态 |
| feedback_form | 反馈页 | 输入问题或建议 | 提交成功/失败 |
| feedback_submitted | 反馈已收到 | 返回我的 | 回到首页 |
| feedback_failed | 反馈提交失败 | 重试或返回 | 保留已输入内容 |
| more_settings | 更多 | 通知/安全/关于 | 进入对应页 |
| delete_account_review | 删除账号说明 | 继续 | 二次确认 |
| emergency_exit | 紧急退出 | 立即退出 | 回登录或锁屏态 |

## 用户端文案口径

优先使用：

- `我的`
- `账号`
- `另一半`
- `反馈`
- `更多`
- `手机号`
- `微信`
- `已绑定`
- `邀请另一半`
- `问题或建议`
- `提交反馈`
- `已收到`
- `退出登录`
- `解除关系`
- `删除账号`

避免：

- `状态机`
- `API`
- `用户对象`
- `数据模型`
- `权限模型`
- `发现流策略`
- `桃桃记住的内容`
- `可见范围`
- `暂未开放`
- `敬请期待`
- `功能建设中`
- `当前模块`
- `开发说明`

## 与当前 MVP 的关系

| 当前能力 | 本分册如何承接 |
| --- | --- |
| 手机号登录 | 账号详情展示脱敏手机号、退出登录 |
| 微信/小程序邀请 | 未绑定时回到邀请半屏，不在我的页重复 |
| 未绑定可使用 | 我的页显示未绑定，并可继续邀请 |
| 三人聊天 | 不在我的页重复展示三人关系 |
| 今天的小事 | 不在我的页处理，仍在聊天胶囊和桃桃页日历 |
| 桃桃改名 | 仍在桃桃页内做，不在我的页重复做 |
| 桃桃日历 | 小事回看和沉淀仍在桃桃页日历，不另做小窝 |
| 未来发现 | 当前不做；分享动作发生时再设计可见范围 |

## 不进入本分册

- 小窝独立首页。
- 三人关系大卡。
- 可见范围独立页。
- 桃桃记住的内容页。
- 桃桃形象生成、装扮、会员。
- 投喂礼物。
- 完整通知中心。
- 复杂客服工单系统。
- 账号换绑和多设备管理。
- 发现信息流本身。

## 验收标准

- 用户能从聊天自然进入我的，并返回聊天。
- 首页不会像传统设置功能列表。
- 首页只保留账号、另一半、反馈、更多。
- 首页不出现可见范围。
- 首页不出现桃桃记住的内容。
- 首页保留轻量反馈入口，方便早期产品收集问题。
- 已绑定和未绑定两种状态都能看懂。
- 解除关系、删除账号都有二次确认。
- 解除关系前必须说明聊天、小事、桃桃和日历的处理方式。
- 设置页不出现内部技术文案。
- 设置页不新增底部 Tab。
- 桃桃改名和日历回看不在设置里重复实现。
- 小窝不作为独立入口出现。
- 删除账号和解除关系不能复用同一个确认文案。
- 反馈提交成功和失败都要有明确状态。

## 视觉方向

视觉上不要像传统设置页，也不要像功能中心。

推荐气质：

- 标题用 `我的`，弱化工具感。
- 首页主入口：账号、另一半、反馈、更多。
- 反馈用轻入口，不做客服中心。
- 账号和另一半只展示必要状态，不做大卡。
- 更多收纳通知、安全、关于。
- 解除关系页要克制、清楚、严肃，不做情绪化装饰。
