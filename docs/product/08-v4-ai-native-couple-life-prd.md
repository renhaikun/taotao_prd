# 桃桃 V4 AI Native Couple Life 产品定义

## 1. 结论

V4 不再是 V3.3 的修补版。V3.3 跑通了“聊天、唤醒、邀请、小程序、小窝”的演示链路，但仍然是 demo 型原型：用户看到很多解释性文案和状态卡，却没有真正经历一个 C 端 AI App 应有的即时理解、生成、反馈、协商和沉淀。

V4 的目标是重新定义产品主干：

```text
用户和另一半共同养一个 AI 小生命。
这个小生命从他们的真实物品、宠物、照片或默认形象中醒来，
在聊天里接住日常小事，
把模糊表达变成可商量的选择，
把完成的小事变成共同记忆，
并在小窝里持续成长。
```

V4 不是“情侣 App + AI 功能”，也不是“AI 聊天工具套情侣皮”。V4 必须让用户第一眼感到：这是一个 AI 时代的新关系产品，而不是小恩爱、情侣空间、打卡工具或情绪评理器。

## 2. 为什么 V3.3 需求评审过不了

### 2.1 首屏没有形成产品理解

当前首屏通过“先聊一句”“需要保存时登录”“它会慢慢认识你们”等文案解释产品，但用户没有立刻看到一个能做事的 AI，也没有明确知道第一步应该做什么。

问题不是文案不够温柔，而是首屏缺少一个真正的工作对象和主动作。成熟 C 端 App 的首屏应当让用户在 5 秒内理解：

- 这是什么。
- 我现在能做什么。
- 做完第一步会发生什么。

### 2.2 唤醒链路是状态跳转，不是产品流程

当前登录后直接进入“桃桃正在看这张照片”，但用户从未上传照片，也没有经历拍照、相册、权限、预览、裁剪、圈选、上传进度、识别确认和生成过程。

这在需求评审里是 P0 断点：核心激活链路不能用按钮跳状态替代。

### 2.3 AI 感没有体现在交互里

当前大量文案在说“桃桃会……”“不会……”“可以……”，但用户没有看到 AI 如何理解、生成、推荐、协商和沉淀。

AI-native 的关键不是页面里出现 AI 词，而是用户动作后系统能产生新的理解和可操作结果：

- 用户说“今晚去公园走走”，系统生成可协商提议。
- 用户上传照片，系统识别来源对象。
- 用户完成一件事，系统生成可确认记忆草稿。
- 两个人意见不同，系统展示差异并给出低压力动作。

### 2.4 聊天没有成为产品引擎

V3.3 说聊天是主入口，但聊天实际仍然是固定 demo 消息和固定卡片。用户输入没有真正触发不同能力。

V4 中，聊天必须成为事件生成器，而不是功能承载页。用户在聊天里自然表达，系统识别场景并生成对应的卡片。

### 2.5 信息架构仍有功能堆叠感

“今天”“留下小事”“看桃桃”“小窝”“生命面板”之间的关系不够清楚。用户不应学习我们的内部结构。

V4 一级结构只保留：

```text
聊天
桃桃
小窝
```

“今天”不再是一级入口，而是聊天里自然出现的动态事件。

### 2.6 用户态混入评审态

“看阿川这边”这样的按钮只能存在于评审态。真实用户不会切换成对方视角。真实用户只会做自己的动作：

- 同意。
- 改一下。
- 晚点答。
- 提醒。
- 撤回。
- 留下。
- 不留下。

### 2.7 小窝没有形成长期留存资产

当前小窝只是记忆列表。V4 的小窝必须承担长期价值：共同事件、共同照片、关系变化、桃桃成长、重要节点、投喂/礼物反馈、可回看的温暖证据。

### 2.8 桃桃主页还不是可操作对象

当前生命面板像说明页。V4 中“桃桃”页必须像一个真实生命对象的主页；如果用户自定义了名字，页面标题和称呼全局跟随用户名称：

- 看它现在状态。
- 改名字。
- 换来源。
- 重新生成形象。
- 处理待办。
- 设置提醒。
- 管理边界。
- 看最近成长。

## 3. V4 一句话定义

桃桃 V4 是两个人共同养的 AI 小生命 App。它从真实生活物品中醒来，在聊天里参与两个人的日常选择、表达和记忆，把关系经营变成轻松、可见、可持续的小互动。

## 4. 产品北极星

### 4.1 北极星体验

```text
共同创建桃桃
-> 桃桃进入两个人的聊天
-> 桃桃接住一件日常小事
-> 两个人轻确认
-> 完成后留下记忆
-> 桃桃在小窝里发生可见变化
```

### 4.2 北极星指标

首发阶段建议使用：

```text
完成共同创建后 7 天内，至少发生 2 次双方参与的桃桃事件的关系数
```

桃桃事件包括：

- 双方共同确认桃桃。
- 双方在同一聊天事件中互动。
- 一件日常小事被协商并完成。
- 一件记忆被双方确认留下。
- 小窝因共同事件产生可见变化。
- 桃桃收到一次正向投喂或礼物反馈。

不把单人聊天次数、上传照片数、打开 App 次数作为北极星。它们是漏斗指标，不代表产品心智成立。

## 5. 产品边界

### 5.1 V4 是什么

- 一个 AI-native 的情侣日常关系产品。
- 一个两个人共同拥有的小生命。
- 一个聊天里发生的关系经营系统。
- 一个共同记忆和轻成长的容器。

### 5.2 V4 不是什么

- 不是情侣空间。
- 不是恋爱任务/打卡 App。
- 不是吵架评理工具。
- 不是 AI 心理咨询。
- 不是 AI 恋人。
- 不是功能工具箱。
- 不是相册时间线。
- 不是会员权益产品。

## 6. 用户心智

### 6.1 第一心智

用户应该这样理解：

```text
我和对方之间多了一个会接住生活的小生命。
```

而不是：

```text
这里有很多情侣功能。
这里有一个 AI 可以帮我们分析关系。
这里能记录纪念日和相册。
```

### 6.2 用户第一次打开时必须感受到

- 有一个小生命正在等我创建。
- 它可以来自我们的真实东西。
- 它之后会进入我和对方的聊天。
- 它不是装饰物，而会参与日常小事。

## 7. V4 信息架构

### 7.1 一级导航

```text
聊天
桃桃
小窝
```

### 7.2 聊天

聊天是主产品，不是功能集合。

用户能做：

- 发消息。
- 上传照片。
- 让桃桃帮忙整理选择。
- 发起一件小事。
- 回应对方的一件小事。
- 保存一件完成的小事。

系统能做：

- 识别用户意图。
- 生成事件卡。
- 跟踪双方状态。
- 在合适时机建议留下记忆。

### 7.3 桃桃

桃桃页是生命对象主页。用户未改名时显示“桃桃”；用户改名后，页面标题、聊天发送者、邀请卡和小窝中的称呼都显示用户自定义名。

用户能做：

- 查看桃桃状态。
- 改名。
- 换来源照片。
- 重生成形象。
- 查看待处理。
- 设置少打扰。
- 管理边界。
- 邀请或重新邀请另一半。

### 7.4 小窝

小窝是长期留存资产。

用户能做：

- 查看共同记忆。
- 查看桃桃成长。
- 编辑记忆摘要。
- 确认或拒绝留下。
- 看共同照片和事件。
- 投喂桃桃后看到小窝变化。

## 9. V4 主流程

### 9.1 首次打开

目标：让用户立刻进入体验，而不是读说明。

首屏结构：

```text
顶部：桃桃品牌
中间：一个待创建的小生命胚胎 / 默认动态形象
底部主动作：创建桃桃
次动作：先聊一句
```

不出现大段介绍。不解释“情侣关系经营”。让用户先动手。

### 9.2 登录

触发时机：

- 创建桃桃。
- 保存聊天。
- 上传照片。
- 邀请另一半。

登录原则：

- 手机号登录简单直接。
- 不把登录页变成产品说明页。
- 登录完成后必须回到触发动作。

### 9.3 创建桃桃

流程：

```text
选择来源
-> 上传/拍照/默认形象
-> 确认来源
-> 生成桃桃
-> 命名
-> 自己确认
-> 邀请另一半确认
```

### 9.4 上传照片

必须有真实界面：

```text
选择来源方式
- 拍一张
- 从相册选
- 先用默认形象

权限态
- 请求相册权限
- 请求相机权限
- 权限拒绝
- 去设置
- 继续用默认形象

上传态
- 预览照片
- 更换照片
- 裁剪
- 手动圈选对象
- 上传中
- 上传失败
- 重新上传
```

### 9.5 识别来源

系统展示 AI 识别结果，但不展示技术术语。

示例：

```text
我看见了：
- 陶瓷杯
- 小猫
- 钥匙扣

主动作：就选这个
次动作：换一个 / 手动输入
```

### 9.6 生成桃桃

生成过程必须可见。

阶段：

```text
正在看照片
正在生成形象
正在做表情
正在准备第一次出现
```

生成结果：

```text
预览动态桃桃
改名字
换一个版本
就用这个
```

异常：

```text
生成失败
生成超时
内容不可用
网络中断
继续等待
重新生成
使用默认形象
```

### 9.7 邀请另一半

App 发起方看到：

```text
邀请未发送
已发送
对方已打开
对方正在看
对方想改
对方晚点答
对方拒绝
邀请过期
双方确认
```

小程序接收方看到：

```text
是谁邀请我
桃桃来自哪里
确认后会发生什么
我可以怎么回应
```

接收方动作：

```text
确认
想改一下
晚点再答
不是这个
```

### 9.8 共同聊天

共同聊天是 V4 的核心日常场景。

用户输入触发事件：

| 用户说 | 桃桃生成 |
| --- | --- |
| 今晚去公园走走？ | 提议卡 |
| 不知道吃什么 | 选择卡 |
| 帮我好好说 | 说法卡 |
| 20:10 提醒我 | 提醒卡 |
| 这个想留下 | 记忆卡 |
| 我有点不开心 | 降温回应，不做评理 |

### 9.9 事件协商

发起方关心：

- 发出去了吗。
- 对方看了吗。
- 对方怎么回应。
- 要不要提醒。
- 能不能撤回。

接收方关心：

- 这是不是承诺。
- 能不能轻改。
- 能不能晚点。
- 回应后对方看到什么。

V4 中，用户不切换视角。系统根据真实用户身份显示对应 UI。

### 9.10 记忆沉淀

记忆不能从“选项”直接生成。必须先发生或被双方确认。

流程：

```text
事件已约好
-> 事件已完成
-> 桃桃建议留下
-> 生成记忆草稿
-> 一方确认
-> 另一方确认
-> 进入小窝
-> 桃桃/小窝发生可见变化
```

## 11. 聊天事件类型

### 11.1 ChoiceCard 选择卡

场景：吃什么、去哪、看什么。

动作：

```text
选一个
换一批
发给对方
晚点再说
```

### 11.2 ProposalCard 提议卡

场景：今晚去公园、周末拍照、一起买东西。

动作：

```text
同意
改时间
改地点
晚点答
撤回
提醒
```

### 11.3 SayBetterCard 好好说卡

场景：想表达但怕生硬。

动作：

```text
温和一点
直接一点
短一点
发给对方
复制
```

### 11.4 ReminderCard 提醒卡

场景：提醒买东西、出门、回消息。

动作：

```text
设提醒
改时间
取消
```

### 11.5 MemoryCard 记忆卡

场景：完成一件小事后。

动作：

```text
留下
改一句
不留下
等对方确认
```

## 12. 文案原则

### 12.1 删除解释型文案

用户端不要出现：

```text
桃桃在听，不会每句话都打断你们
它会慢慢认识你们
这不是承诺
不会变成负担
它只在确实能帮你们变轻时出现
聊天里、底部入口和邀请里都会这样叫它
桃桃是产品名，不会替代它的名字
```

这些是产品逻辑，不是用户文案。

### 12.2 使用动作型文案

推荐：

```text
创建桃桃
上传照片
拍一张
从相册选
就选这个
正在生成
换一个版本
就用这个
邀请阿川
阿川看过了
阿川想改到 21:00
已约好
要留下吗
已放进小窝
```

### 12.3 使用状态型文案

推荐：

```text
上传中
识别中
生成中
等待阿川确认
阿川已打开
阿川晚点再答
小雨已确认
等阿川点头
```

### 12.4 使用结果型文案

推荐：

```text
桃桃醒了
已发给阿川
今晚 20:30
楼下小公园
已提醒
已留下
```

## 13. AI 能力定义

V4 必须展示 AI，而不是解释 AI。

### 13.1 多模态识别

用户上传照片后，系统识别候选对象，并允许用户修正。

### 13.2 生命形象生成

根据来源对象生成动态桃桃，包括：

- 外形。
- 表情。
- 动作。
- 默认语气。
- 可重生成版本。

### 13.3 聊天意图理解

识别用户输入属于：

- 选择。
- 提议。
- 说法优化。
- 提醒。
- 记忆。
- 情绪降温。

### 13.4 关系协商

把一方的模糊表达变成另一方能回应的低压力卡片。

### 13.5 记忆沉淀

从已完成事件中生成短记忆草稿，并等待确认。

### 13.6 安全降温

遇到攻击、羞辱、控制、监控、威胁、自伤、暴力内容时，不评理，不放大冲突，只做降温和安全引导。

## 14. 商业化前置

V4 不做会员制。

早期商业化只埋轻投喂心智：

```text
在两个人完成一件开心小事后，
桃桃可以以拟人方式轻轻要零花钱。
用户随意金额投喂后，
小窝或桃桃产生可见变化。
```

原则：

- 不在低情绪、争执、拒绝、失败时触发。
- 不强制。
- 不做权益墙。
- 不破坏小生命心智。

后续基金/储蓄/渠道推荐属于远期方向，不进入 V4 原型。

## 15. 异常和边界

### 15.1 创建桃桃

必须覆盖：

- 不上传照片。
- 权限拒绝。
- 上传失败。
- 照片无法识别。
- 识别多个对象。
- 生成失败。
- 重生成。

### 15.2 邀请

必须覆盖：

- 对方未打开。
- 对方已打开。
- 对方晚点答。
- 对方想改。
- 对方拒绝。
- 邀请过期。
- 已有关系冲突。

### 15.3 聊天

必须覆盖：

- AI 无法理解。
- 用户撤回。
- 对方未回应。
- 提醒失败。
- 事件取消。
- 事件过期。

### 15.4 记忆

必须覆盖：

- 一方不想留下。
- 一方改摘要。
- 另一方未确认。
- 已确认后隐藏。
- 删除记忆。

## 16. 高保真原型验收标准

V4 高保真原型必须满足：

- 用户首次打开 5 秒内知道主动作是创建桃桃或先聊一句。
- 上传照片链路完整，不再直接跳到识别结果。
- 生成桃桃链路完整，有过程、有失败、有结果、有重生成。
- 聊天输入至少能触发 4 类 AI 事件卡。
- 用户态不出现评审按钮、内部状态、解释产品逻辑的文案。
- 小程序端完整覆盖确认、轻改、晚点、拒绝。
- 桃桃页可操作，不只是展示。
- 小窝能体现长期留存资产。
- 375、390、430 宽度无文字溢出。
- 每个 P0 页面有截图。
- E2E 覆盖首次打开、创建桃桃、邀请、共同聊天、记忆沉淀。

## 17. V4 原型开发建议

不要在 V3.3 上直接堆页面。建议新建 V4 分支或 V4 目录，保留 V3.3 作为证据。

推荐顺序：

```text
Phase 1: 重建信息架构和文案系统
Phase 2: 完整创建桃桃流程
Phase 3: 聊天事件引擎
Phase 4: 邀请与小程序
Phase 5: 桃桃主页与小窝
Phase 6: 评审态与状态矩阵
Phase 7: E2E 和截图证据
```

## 18. 最终判断

V4 的成败不在于页面数量，而在于一个判断：

```text
用户是否相信，这个小生命真的参与了我和另一半的日常。
```

如果用户只看到说明、卡片和功能入口，V4 失败。

如果用户第一次打开就愿意创建桃桃，并在一次日常小事后觉得“它真的在我们之间起作用”，V4 成立。
