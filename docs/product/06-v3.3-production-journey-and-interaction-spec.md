# 桃桃 V3.3 生产级用户旅程与交互规格

## 1. 结论

V3.2 已经跑通主干，但仍然像 MVP demo：它证明了“聊天 + 唤醒 + 邀请 + 小程序 + 小窝”可以串起来，却没有证明真实用户在各种分叉、异常和双方视角里依然觉得顺。

V3.3 的核心不是继续加页面，而是把聊天里的事件设计成生产级对象：

```text
一句话 / 一张图 / 一个时间 / 一个位置
-> 桃桃理解上下文
-> 生成一张可协商、可跟踪、可回收的聊天卡
-> 发起方和接收方看到不同状态
-> 事件完成后进入详情、提醒、小窝或聊天历史
```

这意味着 V3.3 要从“漂亮的主干流程”升级为“真实情侣日常里的状态系统”。高保真原型必须能展示双方视角、异常路径、卡片生命周期和二次入口，否则开发仍然会凭感觉补逻辑。

## 2. V3.2 当前最大缺口

### 2.1 旅程缺口

- 新用户只能被推向手机号登录和唤醒，缺少“我现在只想聊聊、不想上传、不想绑定”的舒适路径。
- 已登录用户重进 App、换设备、恢复历史、临时聊天合并没有用户侧卡片。
- 发起方创建邀请后缺少送达、已打开、对方正在看、过期、撤回、轻提醒。
- 接收方缺少“不马上决定”“晚点再看”“轻改一下会让对方看到什么”的安全感说明。
- 请求修改和拒绝写回 App 后，发起方只看到一句文案，没有明确的换图、改名、重发、收起路径。
- 记忆沉淀缺少双方确认、可见范围、来源消息、编辑/隐藏/拒绝。

### 2.2 状态缺口

当前原型把大量业务压到几个字段里：

- `chatRoom.onboardingStep`
- `awakenSession.inviteStatus`
- `taotaoLife.lifeStatus`
- `miniLinkSession.status`
- `todayScene.memoryDraftStatus`

这些字段能跑 demo，但不能支撑生产：

- 上传照片、权限拒绝、照片不可用、多对象、识别失败、生成失败都没有独立状态。
- `lifeStatus = awake` 出现过早，预览阶段就变成醒来，但真实产品应该双方确认后才醒来。
- 小程序 `miniStep` 是局部 UI 状态，和 `MiniLinkSession` 没有完整一致性。
- 聊天卡片没有 `cardType / cardStatus / owner / version / attempt`，无法表达送达、已读、反提议、过期、撤回。
- “今天”选项一选就进入记忆草稿，跳过了真实情侣之间的协商过程。

### 2.3 UI/文案缺口

- “桃桃”既是产品品牌，又是生命对象名称；如果用户改名叫“小白”，底部 Tab、聊天头像、邀请卡和生命面板会混乱。
- 当前快捷入口仍有功能列表感，能力不是从输入、引用、时间、位置、照片自然浮现。
- 生命面板更像展示页，不像处理待办、边界和二次入口的地方。
- 很多异常路径没有温和文案，用户会不知道自己做了什么、对方会看到什么、能不能撤回。

## 3. V3.3 产品原则

### 3.1 聊天是事件发生地

聊天不是一个消息列表，也不是功能入口集合。聊天是事件被桃桃理解、拆解、协商、完成和沉淀的地方。

所有能力都先以一句自然回应出现，再生成一张状态卡。卡片不是按钮容器，而是一个小事件的生命周期。

### 3.2 同一个事件，不同用户视角

发起方关心：

- 我发出去了没有？
- 对方看到了没有？
- 对方同意、想改、拒绝，还是晚点再说？
- 我能不能提醒、补充、撤回、换方案？

接收方关心：

- 这件事是谁发来的？
- 点了之后会不会直接变成承诺？
- 我能不能轻改、晚点答、不提醒、拒绝？
- 我回应后对方会看到什么？

所以同一个 `ChatCard` 必须按 `viewerRole` 投影成不同 UI。

### 3.3 桃桃只做三件事

- 解释上下文：把一句模糊的话变成更容易理解的小事。
- 维护状态：让双方知道事情走到哪一步。
- 降低压力：允许轻改、晚点、跳过，不催、不站边、不替人决定。

### 3.4 事情发生后，才考虑记忆

“今天晚上去公园”先是一个可协商的约定，不是立刻进入小窝的记忆。只有当事件完成，或双方明确觉得值得留下时，才生成 `MemoryDraftCard`。

### 3.5 关键状态不用 toast

登录失败、授权失败、上传失败、邀请过期、请求修改、拒绝、记忆待确认，都必须成为聊天里的可见卡片或二级详情，不用 toast 承担核心反馈。

## 4. 命名系统

### 4.1 三层名称

| 层级 | 示例 | 用途 |
| --- | --- | --- |
| 产品品牌 | 桃桃 | App 名、登录页、协议、产品来源、商店展示。 |
| 生命默认名 | 桃桃 | 未改名前的生命对象显示名。 |
| 用户自定义生命名 | 小白 | 聊天发送者、生命面板主标题、邀请卡主文案、今天卡里的称呼。 |

内部对象永远叫 `taotaoLife`，不能被用户改名影响。

### 4.2 改名后的显示规则

假设用户把生命名改为“小白”：

| 位置 | 显示 |
| --- | --- |
| App 标题/品牌 | 桃桃 |
| 底部生命 Tab | 小白 |
| 聊天标题 | 小雨、阿川和小白 |
| 桃桃消息发送者 | 小白 |
| 生命面板标题 | 小白 |
| 邀请卡主文案 | 小雨邀请你见见小白 |
| 邀请卡来源角标 | 来自桃桃 |
| 通知 | 小白有一件小事想提醒你 |
| 协议/隐私 | 桃桃 |

### 4.3 文案红线

用户端不要出现：

- 状态机字段、技术名、模型词、内部参考。
- “任务”“房间”“流程”“当前步骤”“开发要点”。
- “AI 心灵信号”“评理”“判定谁对谁错”。
- 对负面场景的投喂、付费、催促。

推荐口径：

- “我先把这件事放轻一点。”
- “这不是承诺，你可以改一下或晚点再答。”
- “我已经告诉小雨你想改哪里。”
- “这件小事还没进入小窝，等你们都点头。”
- “照片以后也可以补，现在可以先用一个默认小东西。”

## 6. 用户旅程地图

| 阶段 | 用户 | 触发 | 用户目标 | 产品反馈 | 下一步 |
| --- | --- | --- | --- | --- | --- |
| 临时聊天 | 新用户 | 首次打开 App | 先感受桃桃，不被迫注册 | “可以先聊，正式保存前再确认手机号。” | 轻聊 / 叫醒 / 今天不提醒 |
| 只想聊 | 新用户 | 输入普通消息 | 不上传、不绑定、不被催 | 桃桃正常回应，降低唤醒提醒频率 | 继续临时聊 / 保存 |
| 登录边界 | 新用户 | 上传、生成、邀请、同步 | 知道为什么要手机号 | 登录卡说明保存范围和回流位置 | 回到原卡片 |
| 恢复历史 | 已登录用户 | 重进 App / 换设备 | 找回进度 | “继续之前的小白 / 保留这次新草稿” | 继续 / 新建草稿 |
| 来源选择 | 发起方 | 开始唤醒 | 选择上传或不用照片 | 上传、默认小物、文字描述、稍后并列 | 识别 / 默认胚胎 |
| 拒绝照片 | 发起方 | 权限拒绝/不想上传 | 不被卡死 | 解释照片以后可补，默认也能先醒来 | 默认生成 / 回聊天 |
| 识别异常 | 发起方 | 图模糊/多对象 | 明确怎么修 | 手动圈选、换图、用默认小物 | 对象确认 |
| 生成预览 | 发起方 | 生成完成 | 判断是否像“我们” | 可改名、换来源、调语气、重新生成 | 发起方确认 |
| 邀请等待 | 发起方 | 发给另一半 | 知道对方是否收到 | 已发送、已打开、正在确认、晚点看 | 提醒 / 撤回 / 改一下 |
| 轻确认 | 接收方 | 微信卡片打开 | 安全地看一眼 | 不会看到聊天；确认后才共同拥有 | 确认 / 轻改 / 晚点 / 拒绝 |
| 修改回流 | 发起方 | 对方轻改/拒绝 | 知道卡在哪里 | 展示对方反馈和可修改项 | 接受修改 / 再提一个 / 收起 |
| 共同聊天 | 双方 | 双方确认 | 开始自然使用 | 桃桃说明不站边、不监控 | 今天事件 / 记忆 |
| 共同协商 | 双方 | “今晚去公园”等输入 | 形成轻约定 | 发起方/接收方不同卡片 | 接受 / 轻改 / 提醒 / 取消 |
| 记忆沉淀 | 双方 | 事件完成后 | 决定是否留下 | 来源、可见范围、待确认方 | 确认 / 编辑 / 隐藏 |
| 二次入口 | 双方 | 进入小窝/生命面板 | 查看待办和边界 | 待确认、少打扰、来源照片、记忆变化 | 回聊天继续 |

## 7. 聊天卡片组件族

| 组件 | 使用时机 | 核心状态 | 完成后去哪里 |
| --- | --- | --- | --- |
| `ContextCueCard` | 桃桃从自然语言、引用、照片、位置、时间里识别到可处理的小事。 | suggested, dismissed, promoted | 收起成聊天提示或升为下一张卡。 |
| `AuthGateCard` | 用户触发正式保存、上传、邀请、同步。 | active, waiting, failed, accepted | 登录后回到原卡。 |
| `SourceChoiceCard` | 唤醒来源选择。 | active, uploading, permission_denied, default_embryo, failed | 进入识别或默认生成。 |
| `RecognitionCard` | 识别来源对象。 | running, single_candidate, multi_candidate, low_confidence, failed | 进入对象确认或换图。 |
| `LifePreviewCard` | 发起方查看生命形象。 | active, editing, pending_self_confirm, accepted | 邀请另一半。 |
| `InviteStatusCard` | 发起方等待另一半。 | sent, delivered, opened, reviewing, countered, accepted, rejected, expired | 共同聊天或修改回收。 |
| `ProposalDraftCard` | 发起方把一句话变成邀约。 | draft, editing, sending, failed | 发出后变发送方卡。 |
| `ProposalSenderCard` | 发起方等待回应。 | sent, delivered, seen, waiting, reminded, countered, accepted, expired | 约定详情 / 聊天摘要。 |
| `ProposalReceiverCard` | 接收方快速回应。 | unread, opened, accepted, countered, snoozed, declined | 同步给发起方。 |
| `CounterProposalCard` | 一方提出轻改。 | counter_pending, accepted, revised, rejected | 合并成最终约定。 |
| `ReminderCard` | 个人提醒或双方提醒。 | off, suggested, on, changed, triggered | 约定详情和个人提醒历史。 |
| `AgreementCard` | 双方确认后。 | agreed, upcoming, started, done, cancelled | 聊天历史 / 今天详情。 |
| `MemoryDraftCard` | 事情完成后建议沉淀。 | draft, pending_partner, needs_edit, confirmed, hidden | confirmed 后进入小窝。 |
| `RecoveryCard` | 异常、中断、恢复。 | failed, retrying, resumable, resolved | 回到原流程。 |

## 8. “今晚去公园”完整双视角

### 8.1 触发

小雨在聊天里说：

```text
今晚去公园走走？
```

桃桃先回复一句：

```text
我先把这件事放成一个小约定，谁都可以改一下。
```

然后生成 `ProposalDraftCard`。

### 8.2 发起方小雨视角

草稿态：

- 标题：今晚去公园走一圈？
- 轻信息：20:30、滨江公园东门、预计 40 分钟。
- 来源：小雨刚刚说“今晚去公园走走？”
- 主动作：发给阿川
- 次动作：改时间、换地点、先不发

等待态：

- 顶部状态：已发给阿川 · 等他点头
- 细状态：19:12 已送达 / 19:14 已查看
- 可操作：补一句、轻轻提醒他、撤回

对方轻改后：

- 状态：阿川想改一下
- 差异：20:30 -> 21:00；滨江公园 -> 楼下小公园
- 主动作：就按他的来
- 次动作：再提一个、先搁置

对方接受后：

- 状态：已约好
- 主动作：20:10 提醒我出门
- 次动作：打开地图、改时间、完成后留下

超时后：

- 状态：阿川还没回应，桃桃先不催
- 动作：晚点提醒我问一次、收起这件事

### 8.3 接收方阿川视角

打开态：

- 标题：小雨想今晚去公园
- 弱提示：桃桃只是在递话，不替你决定。
- 内容：时间、地点、预计多久、天气/距离/回家时间。

快速反馈：

- 可以，20:30
- 晚点 21:00
- 换楼下走一圈
- 今天先不去
- 晚点再答

提醒设置：

- 默认：20:10 提醒我
- 可改：提前 10 分钟 / 出门前 / 不提醒

晚点再答：

- 阿川看到：20:00 我提醒你回应小雨。
- 小雨看到：阿川晚点答，不是没看到。

轻改提交后：

- 阿川看到：已告诉小雨，等她点头。
- 可操作：撤回修改、补一句原因。

### 8.4 桃桃在卡里的角色

桃桃不做第三张大卡，不抢主角。它只在卡片里的小行文承担调停：

- “我先把这件事放轻一点。”
- “这不是承诺，你可以改一下。”
- “我已经告诉小雨你想晚一点。”
- “小雨已经看见你的改动，等她点头。”

## 9. 异常状态设计

### 9.1 不上传照片

场景：

- 用户拒绝相册权限。
- 用户不想上传私密照片。
- 用户不知道该上传什么。

界面：

- 标题：不用照片也可以先叫醒它
- 文案：照片以后可以补。现在可以先用一个默认小东西，让它先学会陪你们说话。
- 主动作：先用默认小物
- 次动作：稍后再传、回聊天

状态：

```text
SourceMedia.status = permission_denied | default_embryo
AwakenSession.status = source_select | generating
```

### 9.2 照片不可用

场景：

- 图像模糊。
- 太暗。
- 没有明确可唤醒对象。
- 有敏感内容。

界面：

- 标题：这张图有点看不清
- 文案：桃桃没有找到适合醒来的小东西。可以手动圈一下，或者换一张更清楚的。
- 主动作：手动圈一下
- 次动作：换图、先用默认小物

### 9.3 生成失败

界面：

- 标题：它刚刚没有醒好
- 文案：照片和选择都还在，不用重新来。
- 主动作：再试一次
- 次动作：稍后继续、先用默认形象

状态：

```text
GenerationJob.status = failed | timed_out | resumable
ChatCard.status = failed
```

### 9.4 邀请过期或撤回

发起方：

- 状态：这条邀请已经过期
- 主动作：重新发一条
- 次动作：改一下再发、收起

接收方：

- 状态：这条邀请已经不能确认
- 主动作：告诉小雨重新发
- 次动作：先回 App

### 9.5 对方已有关系

接收方：

- 标题：你已经有一个共同桃桃
- 文案：为了避免弄乱关系，不能直接加入新的邀请。
- 主动作：查看当前桃桃
- 次动作：稍后处理

发起方：

- 标题：阿川暂时不能加入
- 文案：他那边已经有一个共同桃桃，需要他先处理。
- 动作：稍后提醒、收起邀请

### 9.6 记忆被修改或拒绝

创建方：

- 状态：阿川想改一下这段记忆
- 显示：原文和建议改动。
- 主动作：按他的改
- 次动作：再改一下、不留下

接收方：

- 状态：这件小事还没进入小窝
- 主动作：确认留下
- 次动作：改一句、不留下

## 10. 高保真原型必须新增的画面

### 10.1 默认用户态

- 临时聊天首屏：不强推唤醒，允许先聊聊。
- 登录恢复卡：继续之前的小白 / 保留这次新草稿。
- 来源选择卡：上传、默认小物、文字描述、稍后继续。
- 不上传照片卡：权限拒绝和主动跳过两种。
- 识别异常卡：多对象、低置信、不可用。
- 生成失败卡：重试、稍后、默认形象。
- 命名编辑卡：把“桃桃”改成“小白”，全局显示随之变化。
- 邀请状态卡：已发送、已打开、确认中、轻改、拒绝、过期。
- 共同聊天中的“今晚去公园”发起方视角。
- 共同聊天中的“今晚去公园”接收方视角。
- 对方轻改后的发起方回收卡。
- 约定完成后的记忆草稿卡。
- 记忆待另一半确认和确认后小窝变化。
- 生命面板待办态：待确认、少打扰、来源照片、共同边界。

### 10.2 评审态

评审态需要新增角色/状态切换，而不是只切路由：

- `viewerRole = initiator | receiver`
- `chatCardStatus = draft | sent | delivered | seen | waiting | countered | accepted | expired`
- `sourceMediaStatus = uploaded | permission_denied | unusable | default_embryo`
- `lifeDisplayName = 桃桃 | 小白`

评审态左侧要解释：

- 当前用户是谁。
- 当前卡片是什么类型。
- 这个状态的用户目标是什么。
- 下一步会写入哪个业务对象。
- 完成后卡片回收到哪里。

## 11. 开发改造优先级

### P0：先改信息架构和状态模型

- 引入 `identity`，区分品牌名和生命显示名。
- 引入 `activeCard` 和 `chatCards`，不要继续用一个 `chat-action-card` 承担所有状态。
- 修正 `lifeStatus`：预览不是醒来，双方确认后才 awake。
- 给所有主卡增加 `data-card-type`、`data-card-status`、`data-owner`、`data-viewer-role`。

### P1：补生产级主分支

- 不上传照片 / 默认小物。
- 识别失败 / 多对象 / 生成失败。
- 邀请送达、已打开、请求修改、拒绝、过期。
- 今天协商：发起方、接收方、轻改、接受、提醒、完成。
- 记忆：待对方确认、对方修改、确认后进小窝。

### P2：补二级入口和长期使用

- 生命面板待办和少打扰。
- 小窝空态、待确认态、已确认态。
- 恢复历史和已有关系冲突。
- 小程序身份不一致、邀请被使用、打开 App 失败后的回收。

## 12. 不建议做

- 不把 V3.3 改回独立唤醒向导。
- 不把小程序做成完整 App。
- 不用更多贴纸、爱心、成长值补生命感。
- 不把情侣互动做成任务打卡。
- 不让桃桃评理、站边或裁决。
- 不在争执、拒绝、冷战、明显伤心场景触发投喂或付费。
- 不默认读取聊天记录、相册或微信关系链来制造“懂你们”。

## 13. V3.3 验收标准

V3.3 高保真原型通过的标准：

- 用户 5 秒能理解：这是一个在情侣聊天里帮双方把小事变轻的 AI 生命。
- 新用户可以先聊，不会被一上来逼着上传照片或绑定关系。
- 不上传照片、上传失败、生成失败、邀请失败都能继续，不会卡死。
- 用户改名后，生命名在聊天、Tab、邀请卡、生命面板中一致。
- 同一个今天事件能展示发起方和接收方两个视角。
- 今天事件先协商，再提醒，再完成，最后才进入记忆。
- 发起方能看到送达、已读、对方晚点答、对方轻改、过期。
- 接收方能快速接受、轻改、晚点答、拒绝，并清楚知道对方会看到什么。
- 所有关键状态都有回收位置：聊天历史、约定详情、小窝、生命面板待办。
- 默认用户态不出现技术、流程、状态机和评审文案。
