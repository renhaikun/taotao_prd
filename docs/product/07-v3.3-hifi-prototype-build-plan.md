# 桃桃 V3.3 高保真原型开发计划

## 1. 目标

把当前 V3.2 从“主干 MVP 原型”升级为“生产级高保真体验原型”。这次不是简单补页面，而是重构用户旅程、状态模型和聊天卡片，让开发能直接照着实现真实 App。

V3.3 完成后，原型必须能展示：

- 新用户可先聊天，不被强制上传或绑定。
- 不上传照片、权限拒绝、识别失败、生成失败都有温和分支。
- 品牌名“桃桃”和用户自定义生命名“小白”能同时成立。
- 同一个日常事件有发起方和接收方两个不同视角。
- 聊天卡片能经历送达、已读、等待、轻改、接受、过期、回收。
- 事件完成后先进入约定详情，再决定是否沉淀成小窝记忆。
- 评审态能切换角色和状态，指导开发理解每一步。

必须修正：

- `lifeStatus = awake` 只能在双方确认后出现。
- 小程序状态不能只存在于组件内部，要写回 `miniLinkSession.status`。
- 今天场景不能一选就进入 `memoryDraftStatus=draft`，必须先进入 `waiting_partner / countered / accepted / completed`。

### Phase 2：聊天卡片组件化

把 `TodayActionCard` 和 `AwakenInChatCard` 拆成组件族。

建议新增目录：

```text
src/components/cards/
  AuthGateCard.jsx
  SourceChoiceCard.jsx
  RecognitionCard.jsx
  LifePreviewCard.jsx
  InviteStatusCard.jsx
  ProposalDraftCard.jsx
  ProposalSenderCard.jsx
  ProposalReceiverCard.jsx
  CounterProposalCard.jsx
  AgreementCard.jsx
  MemoryDraftCard.jsx
  RecoveryCard.jsx
```

每张卡必须支持：

```text
data-card-id
data-card-type
data-card-status
data-owner
data-viewer-role
```

卡片视觉规则：

- `suggested/draft`：轻浮层，主动作明确。
- `waiting`：状态轨迹清楚，按钮少。
- `countered`：显示差异，不用大段解释。
- `accepted`：变成稳定约定卡，降低视觉重量。
- `expired/cancelled`：灰度收起，但保留恢复入口。
- `converted_to_memory`：显示小窝入口，不再是主卡。

### Phase 3：补完整唤醒异常

必须新增画面：

- 来源选择：上传照片、默认小物、文字描述、稍后继续。
- 不上传照片：主动跳过和权限拒绝。
- 照片不可用：模糊、太暗、没有明确对象。
- 多对象识别：陶瓷杯、钥匙扣、香薰。
- 生成失败：再试一次、稍后继续、默认形象。
- 命名编辑：从“桃桃”改成“小白”。

关键交互：

- 用户跳过照片后，聊天继续，不让流程中断。
- 使用默认小物后，生命面板需要明确“来源以后可以补”。
- 改名后，底部 Tab、聊天发送者、生命面板、邀请卡全部同步。

### Phase 4：重做邀请与小程序确认

App 发起方必须看到：

- 邀请未发送。
- 已发送。
- 已打开。
- 对方正在确认。
- 对方想改一下。
- 对方拒绝。
- 邀请过期。
- 已撤回。

小程序接收方必须看到：

- 这是谁发来的。
- 确认后会发生什么。
- 不会看到发起方临时聊天。
- 可以稍后再看。
- 可以轻改。
- 可以拒绝且不伤害关系。

需要新增：

- `MiniInviteExpired`
- `MiniIdentityMismatch`
- `MiniExistingBondConflict`
- `MiniLaterResponse`
- `MiniCounterRequest`

### Phase 5：做“今晚去公园”双视角主场景

这是 V3.3 最重要的生产级演示。

发起方小雨视角：

```text
draft
-> sent
-> delivered
-> seen
-> waiting
-> countered
-> accepted
-> completed
-> memory_prompted
```

接收方阿川视角：

```text
unread
-> opened
-> accepted | countered | snoozed | declined
-> waiting_initiator
-> agreed
```

UI 必须不同：

- 发起方看到“有没有送达、有没有看、对方怎么改”。
- 接收方看到“怎么快速答、怎么改、要不要提醒我、对方会看到什么”。
- 桃桃只作为卡片内调停文字出现，不做大段输出。

### Phase 6：重做记忆沉淀

记忆不能从选项直接生成。

新链路：

```text
约定 accepted
-> 事件 completed
-> 桃桃建议留下
-> 创建 MemoryDraftCard
-> 创建者确认
-> 另一半确认
-> 小窝变化
```

记忆草稿必须展示：

- 来源消息。
- 参与者。
- 可见范围。
- 待谁确认。
- 可编辑摘要。
- 对方可以改一句或不留下。

### Phase 7：生命面板变成二次入口

桃桃生命面板不只是展示。

新增模块：

- 待处理：邀请待确认、今天约定待回应、记忆待确认。
- 名字与称呼：展示和编辑生命名。
- 来源照片：更换、补充、删除影响说明。
- 少打扰：减少提醒、只提醒重要事。
- 共同边界：谁能看什么、解除关系、删除数据。

如果生命名是“小白”，页面标题应为“小白”，但底部或角标可保留“来自桃桃”。

### Phase 8：评审态升级

当前评审态只切屏幕。V3.3 要能切角色和状态。

左侧评审面板新增：

```text
viewerRole
activeCardType
activeCardStatus
sourceMediaStatus
lifeDisplayName
miniLinkStatus
memoryStatus
```

评审按钮建议：

- 发起方视角
- 接收方视角
- 不上传照片
- 生成失败
- 邀请已打开
- 对方轻改
- 约定已接受
- 记忆待确认
- 生命名：桃桃 / 小白

## 4. E2E 验收

新增脚本建议：

```text
scripts/e2e-v33-production-journey.mjs
scripts/capture-v33-screenshots.mjs
```

必须断言：

- 默认用户态不出现评审文案。
- `lifeDisplayName = 小白` 时，聊天发送者、Tab、生命面板、邀请卡同步。
- 品牌名“桃桃”仍出现在产品来源和协议类文案，不被改名覆盖。
- 不上传照片后能继续生成默认生命。
- 生成失败能重试或稍后继续。
- 发起方 Proposal 卡有送达、已读、等待、撤回。
- 接收方 Proposal 卡有接受、轻改、晚点、拒绝。
- 轻改后发起方看到 diff。
- 接受后不直接进小窝，而是成为约定。
- 完成后才生成记忆草稿。
- 记忆草稿必须经过双方确认。
- 小程序不显示 App bottom nav。
- 375、390、430 宽度无文字溢出。

## 5. 文案样例

### 临时聊天

```text
你可以先和我说一句。等要保存、叫醒我或邀请另一半时，再确认手机号。
```

### 不上传照片

```text
不用照片也可以先开始。照片以后可以补，我先用一个默认小东西陪你们说话。
```

### 接收方确认前

```text
这不是承诺。你可以点头、改一下，或者晚点再答。
```

### 发起方等待

```text
阿川已经看到了。我先不催他，你可以晚点提醒一次。
```

### 对方轻改

```text
阿川想把时间改到 21:00，地点换成楼下小公园。
```

### 记忆前

```text
这件事还没进小窝。等你们都点头，我再把它收好。
```

## 6. 不进入本轮的范围

V3.3 高保真原型不需要真正接入：

- 真实上传。
- 真实识别。
- 真实 3D 生成。
- 真实微信授权。
- 真实推送提醒。
- 真实持久化。

但必须把这些能力在用户侧的状态和异常表现做出来。
