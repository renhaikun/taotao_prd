# 桃桃 V6 传统高保真状态原型规格

## 目标

V6 不再把聊天页做成“顶部胶囊 + 中部大卡 + 底部推荐”的功能堆叠，而是改成传统生产评审可用的全状态原型：每一步触发后，下一步是什么、谁需要动作、如何回收、从哪里唤醒，都要在 review 面板和截图集中明确展示。

## 核心交互原则

- 顶部胶囊是当前事件的唯一持续上下文。
- 聊天流只放自然消息、桃桃的一句话回应，以及由桃桃气泡承接的轻量事件。
- 桃桃在聊天里等同于第三位参与者：所有桃桃发言必须有桃桃头像、发送者名和自然口吻，不允许做成无头像系统总结卡。
- 大卡只用于创建桃桃、上传/生成、邀请等强流程，不再用于已约好、等待、完成后的聊天事件。
- 底部输入区只显示当前状态的下一步动作，不做固定功能导航。
- 详情 Sheet 承接低频信息：源消息、时间地点、提醒、撤回、留下到小窝。
- 小窝只承接已经进入记忆流程的事件，必须有双方确认状态。

## 展示矩阵

| 状态 | 主展示 | 底部动作 | 回收位置 |
| --- | --- | --- | --- |
| `waiting_partner` | 发起方看顶部胶囊；接收方看回复动作 | 提醒/撤回 或 可以/晚点/晚点再答 | 详情 Sheet |
| `countered` | 发起方看轻改摘要 | 就按这个来 / 再提一个 | 顶部胶囊 |
| `accepted` | 顶部胶囊 + 桃桃自然确认 | 我们回来了 / 提醒一次 / 打开详情 | 详情 Sheet |
| `completed` | 顶部胶囊 + 聊天自然收束 | 留下到小窝 / 打开详情 / 不留下 | 小窝草稿或归档 |
| `memory_prompted` | 小窝草稿 | 我愿意留下 / 改一句 / 不留下 | 等另一半确认 |
| `converted_to_memory` | 小窝已确认记忆 | 继续聊天 | 小窝 |

## 用户端禁区

用户端不出现以下类型文案：

- 内部状态名、技术字段、原型说明。
- “AI 心灵信号”“今日行动”“房间”等破坏产品心智的词。
- “桃桃会在合适的时候接一句”“它会怎么陪”等解释性能力文案。
- “桃桃已经接上这件小事”“在微信里选了”等第三人称系统通知式文案。
- 同一事件在顶部、聊天大卡、底部推荐中重复展示。

## 聊天内事件职责

- 顶部胶囊：只回答“现在有哪件事正在进行、状态是什么”，不承担解释和按钮堆叠。
- 桃桃气泡：只回答“桃桃作为第三个人怎么接住这句话”，可以轻轻提出下一步，但不写系统报告。
- 底部动作：只回答“此刻用户能点什么”，不复述事件标题、时间和状态。
- 详情 Sheet：只在用户主动打开时承接完整记录、低频动作和小窝回收。
