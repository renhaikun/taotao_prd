# 桃桃 V2 M0/M1 范围

## 1. 切分原则

V2 的版本切分不以功能数量为准，而以“生命感闭环”是否成立为准。

M0 必须惊艳但可上线：

- 惊艳来自首次唤醒、动态生命感、双方确认、今日现场，不来自功能堆叠。
- 可上线意味着每个核心状态有真实 UI、真实数据结构、异常处理和可继续路径。
- 不做重功能，不做复杂游戏数值，不做深商城，不做全量 3D，不做传统情侣 App 的大而全。

M1 承接留存和深度：

- 强化记忆小窝、更多今日现场、更多个性成长。
- 加强 App + 小程序联动。
- 增加付费和资产，但仍保持轻。

## 2. M0 产品目标

M0 要证明：

1. 用户能理解并愿意发起“唤醒你们的小东西”。
2. 另一半能理解并完成共同确认。
3. 桃桃醒来后能立刻进入今日现场。
4. 三方群聊不是 AI 问答页。
5. 至少一件共同小事能进入记忆小窝并带来可见变化。

M0 不追求：

- 全部情侣功能覆盖。
- 长期相册体系。
- 大量装扮和商城。
- 复杂任务和成就。
- 完整实时 3D。
- 关系咨询能力。

## 3. M0 必做范围

### 3.1 App

| 模块 | M0 必做 | 验收标准 |
| --- | --- | --- |
| 首次进入 | seed 态小东西、登录、恢复会话。 | 第一屏不是功能首页；用户知道要唤醒。 |
| 关系绑定 | 创建邀请、打开邀请、接受/拒绝、绑定成功。 | 邀请状态可见；异常不靠 toast。 |
| 首次唤醒 | 来源选择、上传/默认胚胎、AI 识别、对象确认。 | 可完成真实流程；失败可换图或默认。 |
| 桃桃生成 | 生成一个可用候选、动态预览、名字和性格。 | 不要求实时 3D，但必须有动效和一致形象。 |
| 双方确认 | 发起方确认、另一半确认、请求调整、拒绝。 | 未双方确认不能进入完整共同体验。 |
| 醒来完成 | 第一次睁眼、进入今日现场。 | 不停在生成结果页。 |
| 今日现场 | 照片现场、桃桃动态在场、一张今日决定卡、一个主行动。 | 根据状态推荐，不做功能宫格或信号列表。 |
| 三方群聊 | 她、他、桃桃三类消息、基础发送、桃桃回应、行动卡。 | 看起来是三方在场，不像单人问 AI。 |
| 记一件小事 | 从今日现场或群聊生成草稿、保存、另一半确认。 | 可写入记忆小窝。 |
| 记忆小窝 | 展示已确认小事、小窝初始变化。 | 小窝变化和事件对应。 |
| 安全边界 | 基础攻击/羞辱/监控/自伤/暴力拦截。 | 给降温卡或替代动作。 |
| 账号与数据 | 核心对象和状态落库。 | App 重启后能恢复到正确状态。 |

### 3.2 小程序

| 模块 | M0 必做 | 验收标准 |
| --- | --- | --- |
| 微信授权 | 识别用户。 | 可恢复邀请上下文。 |
| 邀请打开 | 展示发起人、来源对象、桃桃预览。 | 用户知道确认的是什么。 |
| 另一半确认 | 接受、请求调整、拒绝。 | 状态同步到 App。 |
| 已绑定轻入口 | 展示桃桃轻状态和一个今日动作。 | 不复制完整 App。 |
| 回 App | 带上 `coupleId/sessionId/sourceScene`。 | 回 App 后状态不断。 |

小程序 M0 可以不做完整生成；生成主要在 App。若用户从小程序确认唤醒，只确认已有候选。

### 3.3 AI 与后端

| 能力 | M0 实现方式 |
| --- | --- |
| 来源对象识别 | 支持上传图片，输出候选对象；失败可默认胚胎。 |
| 桃桃形象生成 | 生成 1 个主候选 + 必要动作资产；允许异步生成。 |
| 桃桃口吻 | 统一人格和话术层，不暴露模型。 |
| 场景识别 | 覆盖唤醒、确认、今日现场、群聊、记忆、安全。 |
| 安全检查 | 输入前置检查 + 输出复检。 |
| 状态存储 | `CoupleBond`、`AwakenSession`、`TaotaoLife`、`TodayScene`、`NestMemory`。 |

## 4. M0 可降级项

| 项目 | 理想态 | M0 可接受降级 |
| --- | --- | --- |
| 3D 动态 | 实时 3D/可交互。 | 2.5D 动态图、短视频、Lottie 或序列帧。 |
| 多候选生成 | 3 个候选可切换。 | 1 个候选 + 轻量重生成。 |
| 来源识别 | 自动识别对象和语义。 | 自动识别失败时允许手动圈选。 |
| 小窝空间 | 完整可装修空间。 | 固定小窝舞台 + 事件触发的小变化。 |
| 三方群聊 | 流式回复和丰富行动卡。 | 基础消息 + 关键行动卡。 |
| 记忆 | 图片、文字、地点、情绪等完整结构。 | 标题、文字/图、桃桃一句话、双方确认。 |
| 小程序 | 轻现场和确认全覆盖。 | 邀请确认 + 回 App。 |

降级不能破坏的底线：

- 不能把唤醒改成选头像。
- 不能取消双方确认。
- 不能把今日现场做成功能宫格。
- 不能把群聊做成 AI 问答。
- 不能只用静态文案代替桃桃生命状态。

## 5. M0 不做范围

明确不进入 M0：

- 会员订阅。
- 完整商城。
- 深度小窝装修系统。
- 装扮背包和稀有度。
- 多个 AI 角色。
- 复杂关系测评。
- 谁对谁错评理。
- 定位追踪。
- 自动抓取微信聊天。
- 完整相册时间线。
- 纪念日工具集合。
- 每日打卡和惩罚机制。
- 复杂理财、愿望罐、外部导流。
- 大规模 UGC 社区。

## 8. M0 验收用例

M0 必须跑通以下端到端用例：

1. A 打开 App，看到 seed 态，登录后上传照片，识别对象，生成桃桃候选。
2. A 修改桃桃名字和性格，确认后发给 B。
3. B 从小程序打开邀请，看到同一个来源对象和桃桃候选，点击确认。
4. 双方确认后，App 中桃桃第一次睁眼，并进入今日现场。
5. 今日现场推荐一件轻行动，A 开始，B 确认，行动完成。
6. 桃桃把行动转成一件小事，A 保存，B 确认。
7. 记忆小窝出现对应小变化。
8. A/B 进入三方群聊，双方都发言，桃桃回应并保持人格一致。
9. 上传失败、生成失败、邀请过期、消息失败、安全拦截都有可见状态和下一步。

## 9. M1 范围

M1 目标是让桃桃从“被唤醒”进入“持续共同生活”。

### 9.1 体验增强

- 更多今日现场类型：吃什么、周末做什么、送礼灵感、好好说、安抚一下。
- 更丰富的三方群聊行动卡。
- 更完整的记忆详情。
- 小窝时间轴和成长变化。
- 桃桃心情和轻提醒。
- 安静模式和提醒偏好。

### 9.2 生成与动态增强

- 多候选生成。
- 局部调整：耳朵、颜色、挂件、动作。
- 更多动作：开心、困、想你们、收到礼物、记住小事。
- 更稳定的跨页面资产包。
- 来源照片删除和形象保留策略完整化。

### 9.3 小程序增强

- 小程序轻群聊预览。
- 小程序轻吃什么。
- 小程序确认记忆。
- 小程序查看一条小窝变化。
- 更稳定的回 App deep link。

### 9.4 轻商业化

M1 可以开始做轻投喂，但必须满足：

- 只在正向事件后出现。
- 有“不投喂也没关系”。
- 不在争执、冷战、明显伤心时出现。
- 投喂后有小窝变化，不只是支付成功。

M1 不建议直接做会员订阅。

## 10. M1 不做或谨慎做

| 项目 | 原因 |
| --- | --- |
| 会员订阅 | 容易破坏小生命和共同照料心智。 |
| 复杂商城 | 会让小窝变成消费页。 |
| 情侣评分 | 与安全和低压力原则冲突。 |
| 关系诊断报告 | 会引导评理和焦虑。 |
| 自动读取聊天记录 | 授权和信任成本过高。 |
| 社区广场 | 早期会稀释双人私密体验。 |
| 重度游戏化 | 会让真实关系退居其次。 |

## 11. M0 到 M1 的判断门槛

进入 M1 前，M0 至少满足：

- 唤醒开始率达到可观察规模。
- 从发起唤醒到另一半确认的漏斗可用。
- 桃桃醒来后 24 小时内有今日现场使用。
- 三方群聊中出现双方参与，而不是单人使用。
- 至少一部分关系完成记忆确认。
- 异常状态没有造成大面积卡死。
- 用户反馈没有把产品理解为传统情侣工具或 AI 助手。

只有这些成立，M1 才扩展更多场景和轻商业化；否则优先修首次唤醒、共同确认和今日现场。
