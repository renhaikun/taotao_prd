# 桃桃 V3.2 Chat-first 产品流程

## 1. V3.2 定义

V3.2 的产品本体是 `聊天`，不是唤醒页、邀请页、任务页或功能首页。用户默认打开 App 后，只应该看见真实 C 端产品界面；技术字段、流程解释和开发提醒只出现在产品文档、测试代码和 `?mode=review` 评审态中。

用户默认打开 App 后进入一个可继续的聊天。未登录用户进入临时聊天；完成手机号注册/登录后，临时聊天升级为正式聊天，并允许正式唤醒桃桃、创建邀请、沉淀记忆和跨端同步。

V3.2 保留 V2 的生命对象、双方确认和 App + 小程序双端逻辑，但首体验从“先看一个唤醒流程”调整为：

```text
打开 App
-> 进入聊天
-> 桃桃以聊天式 onboarding 引导用户认识、命名和唤醒
-> 手机号注册/登录
-> 正式创建 AwakenSession
-> 邀请另一半
-> 双方在 App/小程序确认
-> 桃桃成为共同聊天里的生命对象
-> 记忆和生命状态持续沉淀
```

## 2. 产品原则

### 2.1 Chat-first

- 聊天是默认首页，也是所有状态回流的容器。
- 唤醒、邀请、今天场景、记忆确认都以聊天中的卡片和对话推进。
- 页面切换不能让用户感觉离开了桃桃；即使进入上传、登录、小程序确认，也要能回到同一段聊天上下文。
- 聊天不是 AI 问答页，桃桃不是客服助手。聊天里始终表达“我、另一半、桃桃”三方在场。

### 2.2 登录先于正式能力

- 未登录可以体验轻量临时聊天，但不能上传正式照片、调用生成、创建邀请、写入共同记忆或同步到另一端。
- 手机号注册/登录发生在正式唤醒与邀请之前，不做上传后再补登录的生产风险路径。
- 登录不是营销页，也不是账号中心；它是把临时聊天保存下来的状态边界。

### 2.3 唤醒是聊天式 onboarding

- “唤醒桃桃”不再是独立向导页，而是一组聊天内连续对话和状态卡。
- 桃桃通过提问、轻选择、上传卡、识别卡、确认卡引导用户完成唤醒。
- 每一步只推动一个决定：先认识你，再选择来源，再确认对象，再生成生命形象，再邀请另一半。

### 2.4 桃桃页是生命面板

- 桃桃页不再承载邀请状态主线。
- 桃桃页展示生命状态、来源、人格、能量、动作、记忆成长和共同照料边界。
- 邀请、确认、登录中断等流程状态优先回到聊天内处理。

### 2.5 小程序是微信卡片打开的独立轻端

- 小程序不是 App 镜像，也不是网页落地页。
- 小程序从微信邀请卡片打开，独立完成授权、邀请查看、确认、轻回应和回 App。
- 小程序必须理解当前 `MiniLinkSession`，不能要求用户重新解释上下文。

### 2.6 视觉克制但有生命感

- 整体视觉偏优雅、克制、INS 风生活摄影感。
- 核心画面优先使用真实生活照片、柔和光线、自然留白和低密度信息。
- 不做传统情侣 App 的粉色堆叠、贴纸化爱心、任务打卡感。
- 不做花哨 AI 炫技，不展示模型、参数、内部生成指令或复杂科技纹理。
- 桃桃的生命感来自动作、停顿、语气、光感和聊天上下文，不靠过量特效。

### 2.7 人格型助手参考

V3.2 可以参考蚂蚁阿福这类人格型 AI App 的方向：它不是把能力平铺成工具箱，而是先让用户知道“这个助手是谁、能陪我做什么、边界在哪里”，再把问答、陪伴和服务组织成可理解的场景。

对桃桃的落地要求：

- 第一心智是“桃桃在场”，不是“情侣工具集合”。
- 所有能力先由桃桃用一句自然回应承接，再递出一张场景卡。
- 场景卡只解决一个眼前动作，例如登录保存、选择来源、邀请确认、记一件小事。
- 桃桃要有边界：不站边、不监控、不替代双方决定，不把高风险内容包装成温情记忆。
- 用户端界面不出现竞品名、内部参考、模型能力炫耀或功能说明式长文。

## 3. 核心对象与状态

| 对象 | V3 说明 | 关键字段 |
| --- | --- | --- |
| `ChatRoom` | V3 默认首页和产品本体。 | `roomId`、`roomType`、`userId`、`coupleId`、`taotaoId`、`roomStatus`、`lastContext` |
| `TemporaryRoom` | 未登录前的临时聊天。 | `tempRoomId`、`deviceId`、`expiresAt`、`onboardingProgress`、`handoffPayload` |
| `UserProfile` | 登录后的用户身份。 | `userId`、`phone`、`nickname`、`avatarUrl`、`platformBindings` |
| `CoupleBond` | 双方关系和共同拥有权。 | `coupleId`、`memberA`、`memberB`、`bondStatus`、`createdBy`、`confirmedBy` |
| `AwakenSession` | 聊天式唤醒会话。 | `sessionId`、`roomId`、`sourceType`、`sourceMediaId`、`recognitionStatus`、`generationStatus`、`confirmStatus` |
| `TaotaoLife` | 桃桃生命对象。 | `taotaoId`、`lifeStatus`、`name`、`sourceObject`、`persona`、`motionSet`、`energyState`、`memoryCount` |
| `MiniLinkSession` | 微信卡片和小程序轻端会话。 | `miniSessionId`、`inviteToken`、`targetAction`、`status`、`openAppPayload` |
| `NestMemory` | 共同确认后沉淀的小事。 | `memoryId`、`sourceRoomId`、`sourceMessageId`、`confirmStatus`、`visibility`、`nestEffect` |

### 3.1 聊天状态 `roomStatus`

| 状态 | 含义 | 用户侧表现 |
| --- | --- | --- |
| `temporary` | 未登录临时聊天。 | 可聊天式了解桃桃，可保存意向，正式能力前要求登录。 |
| `auth_required` | 用户触发正式能力。 | 登录卡插入当前对话，不丢上下文。 |
| `single_onboarding` | 已登录，未绑定，正在聊天式唤醒。 | 桃桃继续引导来源选择、命名、生成和邀请。 |
| `invite_pending` | 已创建邀请，等待另一半。 | 聊天内展示邀请进度、微信卡片入口、提醒和稍后继续。 |
| `partner_reviewing` | 另一半在小程序或 App 查看。 | 聊天内显示对方正在查看，不开放完整共同体验。 |
| `couple_ready` | 双方确认关系，桃桃已醒来或可继续唤醒。 | 三方聊天成立，开始共同对话和今天场景。 |
| `active` | 日常使用。 | 聊天、生命面板、记忆和轻行动持续流转。 |
| `paused` | 少打扰或无新事件。 | 桃桃安静在场，不连续催促。 |

### 3.2 聊天式唤醒状态 `onboardingStatus`

| 状态 | 产品含义 | 聊天内组件 |
| --- | --- | --- |
| `intro` | 桃桃第一次出现，解释自己不是工具。 | 桃桃轻问候 + 一个继续按钮。 |
| `name_user` | 认识当前用户。 | 昵称卡，可跳过。 |
| `source_select` | 询问桃桃可以从哪里醒来。 | 来源选择卡：照片、宠物、礼物、玩偶、纪念物、默认胚胎。 |
| `auth_gate` | 正式唤醒前要求手机号登录。 | 登录卡，说明保存聊天和生成需要身份。 |
| `source_upload` | 已登录后上传或选择来源。 | 上传卡、相册权限、默认胚胎入口。 |
| `recognizing` | 识别来源对象。 | 识别中消息、对象轮廓、取消/稍后继续。 |
| `object_confirm` | 确认要唤醒的对象。 | 候选对象卡、手动圈选、换图。 |
| `generating` | 创建桃桃生命形象。 | 生成中卡，保留聊天输入区。 |
| `life_preview` | 展示初始桃桃。 | 动态预览、名字、性格、轻编辑。 |
| `self_confirmed` | 发起方确认。 | 邀请另一半卡。 |
| `partner_pending` | 等另一半确认。 | 邀请进度卡、微信分享卡状态。 |
| `awake` | 双方确认完成。 | 桃桃第一次醒来消息 + 进入共同聊天。 |

## 4. App 全链路

### 4.1 首次打开 App

```text
打开 App
-> 创建 TemporaryRoom
-> 进入聊天
-> 桃桃轻问候
-> 用户可回复或点击继续
-> 触发正式能力前插入登录卡
-> 手机号注册/登录
-> TemporaryRoom 合并为 ChatRoom
-> 继续聊天式唤醒
```

首屏要求：

- 第一屏必须是聊天，不是营销页、功能首页或独立唤醒页。
- 聊天顶部只展示轻身份：桃桃、当前聊天状态、必要的返回/更多。
- 底部输入区默认可用，但未登录时不允许触发正式上传、生成和邀请。
- 桃桃第一轮话术要短，重点建立生命感，不解释全部功能。
- 如果用户退出再回来，恢复到同一条 onboarding 进度。

### 4.2 手机号注册/登录

```text
用户点击正式唤醒 / 上传 / 邀请
-> 聊天内出现登录卡
-> 输入手机号
-> 验证码
-> 轻资料
-> 创建或恢复 UserProfile
-> 绑定 TemporaryRoom 历史
-> 回到触发登录前的聊天卡片
```

登录卡要求：

- 清楚说明登录是为了保存桃桃、生成资产、邀请另一半和跨端同步。
- 登录失败时保留手机号、验证码输入状态和原聊天上下文。
- 已注册用户登录后应恢复历史 `ChatRoom`、`CoupleBond` 和 `TaotaoLife`。
- 新用户登录后继续当前临时聊天，不跳转到空首页。

### 4.3 聊天式唤醒

```text
ChatRoom(single_onboarding)
-> 来源选择
-> 上传/默认胚胎
-> 识别对象
-> 用户确认对象
-> 生成桃桃生命形象
-> 预览并轻编辑
-> 发起方确认
-> 创建邀请
-> 等另一半
```

交互要求：

- 每个步骤以聊天消息 + 状态卡推进，卡片完成后沉入聊天历史。
- 用户可以发送自然语言，系统根据当前状态转换为下一步动作或澄清问题。
- 上传、识别、生成失败不能把用户带离聊天；失败卡必须给重试、换图、默认胚胎、稍后继续。
- 生成结果是 `TaotaoLife` 初始形态，不是头像选择器。
- 发起方确认后，完整共同体验仍然锁定，直到另一半确认。

### 4.4 邀请另一半

```text
发起方确认桃桃预览
-> 创建 CoupleBond 草稿
-> 创建 MiniLinkSession
-> 生成微信邀请卡
-> 用户分享给另一半
-> 聊天进入 invite_pending
```

邀请卡必须展示：

- 发起方昵称。
- 来源对象和桃桃预览。
- “想和你一起叫醒桃桃”的共同语境。
- 当前确认进度。
- 微信分享、复制链接、稍后提醒。

邀请等待状态：

- 聊天内可继续和桃桃轻聊天，但不生成共同记忆。
- 桃桃可以解释等待状态，但不能催促或制造压力。
- 发起方可以调整预览；调整后需要刷新邀请确认内容。
- 邀请过期、被拒绝、请求修改都以聊天卡片回流。

### 4.5 另一半从 App 打开

```text
另一半打开 App 链接
-> 手机号注册/登录
-> 恢复 inviteToken
-> 进入同一个 ChatRoom 的确认视图
-> 查看来源对象、桃桃预览、发起方
-> 接受 / 想改一下 / 不是这个
-> 状态写回发起方聊天
```

规则：

- 另一半必须完成身份识别后才能确认关系。
- 请求修改不删除来源和生成资产，回到发起方聊天的 `life_preview`。
- 拒绝后保留发起方个人草稿，可换图、改名或重新邀请。
- 双方确认后，聊天升级为 `couple_ready`，桃桃第一次醒来。

### 4.6 醒来后的日常聊天

```text
TaotaoLife(awake)
-> ChatRoom(couple_ready)
-> 桃桃第一条醒来消息
-> 递出一个轻松的共同动作
-> 双方聊天或轻确认
-> 可沉淀为记忆
-> 桃桃生命面板同步更新
```

日常聊天要求：

- 默认入口仍然是聊天。
- “今天”不再是独立首页或任务页，而是聊天中的主行动卡或沉浸生活场景。
- 聊天可承载：普通陪伴、今天做什么、吃什么、好好说、记一件小事、确认记忆、小窝变化。
- 底部 Tab 可以存在，但不能让功能宫格抢走聊天心智。

## 5. 微信小程序全链路

### 5.1 从微信卡片打开

```text
微信邀请卡
-> 打开小程序
-> 读取 inviteToken
-> 创建 MiniLinkSession
-> 展示轻预览
-> 微信授权
-> 进入确认页
```

轻预览在授权前可展示：

- 发起方昵称。
- 桃桃模糊或轻量预览。
- 一句共同邀请文案。
- 授权用途说明。

授权前不能展示完整共同信息、不能确认关系、不能写入状态。

### 5.2 小程序确认

```text
微信授权成功
-> 绑定或创建 UserProfile.platformBindings
-> 校验 inviteToken
-> 展示来源对象、桃桃预览、名字、性格、发起方
-> 接受 / 想改一下 / 不是这个
-> 写回 CoupleBond 和 AwakenSession
-> 展示下一步
```

确认结果：

| 操作 | 写入状态 | 下一步 |
| --- | --- | --- |
| 接受 | `confirmStatus.partner = accepted`，若双方都接受则 `lifeStatus = awake`。 | 展示完成态，打开 App 或先进行一个轻回应。 |
| 想改一下 | `confirmStatus.partner = needs_edit`。 | 小程序展示已反馈，App 聊天回到预览编辑卡。 |
| 不是这个 | `confirmStatus.partner = rejected`。 | 小程序展示温和结束态，App 保留发起方草稿。 |

### 5.3 小程序轻回应

双方确认后，小程序可以提供一次轻回应，而不是复制完整 App。

可做：

- 给桃桃说第一句话。
- 对今日主行动做一个轻选择。
- 确认一条记忆草稿。
- 查看一条小窝变化。
- 打开 App 继续聊天。

不可做：

- 完整图片上传和生成。
- 大量聊天历史。
- 完整生命面板编辑。
- 完整记忆时间线。
- 设置、解绑和数据删除等高风险操作。

### 5.4 回 App

```text
小程序完成确认/轻回应
-> 创建 openAppPayload
-> 打开 App
-> App 读取 miniSessionId/coupleId/roomId/sourceAction
-> 回到对应 ChatRoom
-> 插入 mini_return_card
```

回流要求：

- 打开成功后，App 的聊天显示小程序刚完成的动作。
- 打开失败时，小程序提供复制链接、重新打开、稍后提醒。
- 回流 token 过期时，App 要能通过登录身份恢复最近的 `ChatRoom`。

## 6. 注册登录中断与回流

### 6.1 中断场景

| 场景 | 必须保留 | 回流位置 |
| --- | --- | --- |
| 未登录触发上传 | 临时聊天、来源选择意图。 | 登录后回到上传卡。 |
| 未登录触发生成 | 已选来源、上传结果或默认胚胎。 | 登录后创建 `AwakenSession` 并继续识别/生成。 |
| 未登录触发邀请 | 桃桃预览、名字、性格。 | 登录后创建邀请卡。 |
| 小程序授权失败 | inviteToken、发起方、轻预览。 | 重新授权后回到确认页。 |
| App 被系统杀掉 | 当前 roomId、onboardingStatus、未发送输入草稿。 | 重启后恢复聊天。 |
| 网络失败 | 当前操作 payload、失败原因、重试次数。 | 原卡片内重试。 |

### 6.2 合并规则

- `TemporaryRoom` 合并到登录用户时，按 `tempRoomId + deviceId + expiresAt` 校验。
- 如果登录用户已有活跃 `ChatRoom`，优先提示“继续之前的桃桃”或“保留这次新聊天草稿”，不能静默覆盖。
- 如果用户已有 `CoupleBond`，不能接受新的绑定邀请，必须提示当前关系状态和可选处理路径。
- 小程序微信身份与手机号身份合并时，以手机号登录为主账号，微信 `openId/unionId` 作为平台绑定。

### 6.3 用户体验底线

- 任何登录、授权、回流中断都不能丢失用户刚刚做出的选择。
- 不用 toast 承担关键错误，关键状态必须成为聊天内可见卡片。
- 失败后必须有下一步：重试、换方式、稍后继续、复制链接或回到聊天。

## 7. 桃桃生命面板

桃桃生命面板是二级页，用于查看和轻管理生命对象。它不替代聊天，也不承接邀请主流程。

### 7.1 入口

- 聊天顶部桃桃头像。
- 底部 Tab 的“桃桃”。
- 记忆或小窝变化卡片中的“看看桃桃”。

### 7.2 面板结构

| 区域 | 内容 | 可操作 |
| --- | --- | --- |
| 生命状态 | 当前心情、能量、在场状态、最近一次回应。 | 少打扰、提醒偏好。 |
| 形象与动作 | 主形象、待机/开心/思考/醒来动作。 | 查看动作，M1 可轻编辑。 |
| 来源 | 来源对象、来源照片状态、唤醒时间。 | 更换展示、删除来源照片说明。 |
| 人格 | 名字、称呼、说话方式、性格摘要。 | 双方确认后改名或调整语气。 |
| 记忆成长 | 已确认记忆数、最近小窝变化、待确认记忆。 | 跳转到对应记忆卡。 |
| 共同边界 | 数据可见范围、通知、解除关系、删除数据。 | 高风险操作二次确认。 |

### 7.3 状态要求

- 未醒来时，生命面板展示胚胎状态和当前唤醒进度，但主动作回到聊天继续。
- 待另一半确认时，生命面板只展示确认进度，不把邀请页搬进来。
- 已醒来后，面板重点是生命成长，不是设置中心。
- 删除来源照片时，必须说明是否保留已生成形象和已有记忆。

## 8. 记忆沉淀

V3 的记忆从聊天中自然生成，不从独立表单开始。

### 8.1 生成路径

```text
聊天消息 / 今天场景卡 / 小程序轻回应
-> 桃桃识别可沉淀的小事
-> 生成记忆草稿卡
-> 创建者确认
-> 另一半确认
-> 写入 NestMemory
-> 桃桃生命面板和小窝同步变化
```

### 8.2 记忆草稿

记忆草稿必须包含：

- 标题。
- 来源聊天和来源消息。
- 时间。
- 参与者。
- 用户原话或行动摘要。
- 桃桃一句话。
- 可见范围。
- 待确认方。

### 8.3 记忆状态

| 状态 | 含义 | 展示位置 |
| --- | --- | --- |
| `draft` | 创建者可编辑。 | 聊天记忆草稿卡。 |
| `pending_partner` | 等另一半确认。 | 聊天待确认卡、生命面板待办。 |
| `needs_edit` | 另一半请求调整。 | 回到草稿卡。 |
| `confirmed` | 进入共同记忆。 | 聊天历史、生命面板、记忆小窝。 |
| `hidden` | 双方选择隐藏。 | 仅在边界/管理入口可见。 |
| `deleted` | 删除后保留必要审计。 | 用户侧不可见。 |

### 8.4 记忆边界

- 单人私密输入不能默认变成共同记忆。
- 争执、羞辱、监控、自伤、暴力等高风险内容不生成温情记忆。
- 另一半未确认前，不进入共同小窝，不触发生命成长反馈。
- 桃桃不夸大总结，不做关系评判，只保留具体小事。

## 9. 导航 IA

### 9.1 App 一级结构

| 入口 | V3 含义 | 默认落点 |
| --- | --- | --- |
| 聊天 | 产品本体，所有流程的默认容器。 | 当前 `ChatRoom`。 |
| 小窝 | 已确认共同小事和小窝变化。 | 记忆列表/小窝轻舞台。 |
| 桃桃 | 生命面板。 | `TaotaoLife` 当前状态。 |

导航规则：

- App 启动默认进入“聊天”。
- 未登录默认进入临时聊天，不展示完整功能宫格。
- 聊天页本身不显示底部导航，避免 composer 和 Tab 并存。
- 唤醒、邀请、登录回流完成后都回到“聊天”。
- 记忆和桃桃页的主动作应回到聊天继续完成。
- 关系与边界管理后续可放在“小窝”或“桃桃”的二级设置里，M1 不做独立一级入口。
- “今天”不是一级入口；它是聊天里出现的沉浸生活场景。

### 9.2 小程序结构

| 页面 | 用途 |
| --- | --- |
| 邀请轻预览 | 授权前理解邀请。 |
| 微信授权 | 识别用户并保留 inviteToken。 |
| 确认页 | 查看并确认桃桃候选。 |
| 轻回应页 | 双方确认后完成一次低成本互动。 |
| 回 App 页 | 打开 App 或复制链接。 |

小程序不设置完整底部 Tab，不复制 App 的生命面板和记忆系统。

## 10. 生产验收清单

### 10.0 用户态与评审态

- 默认 `/` 只展示真实 App 用户界面，不出现左侧流程面板、技术字段、开发说明或原型说明。
- `?mode=review` 才展示流程说明，用于产品、设计和开发对齐。
- 用户端文案不出现“房间”“任务”“当前步骤”“开发要点”“route”“surface”等内部语汇。
- 技术字段、状态机和实现说明必须写在文档、测试和评审态里，不能写到用户界面。

### 10.1 产品链路

- App 首次打开默认进入聊天，不进入传统首页。
- 未登录临时聊天可用，但正式上传、生成、邀请、记忆写入前必经手机号登录。
- 登录完成后回到触发登录前的同一张聊天卡片。
- 聊天式唤醒能完整走通：来源选择、上传/默认胚胎、识别、对象确认、生成、预览、发起方确认。
- 邀请创建后，发起方聊天能看到邀请状态和另一半回流结果。
- 另一半从 App 或小程序确认后，双方状态同步，桃桃进入醒来状态。
- 桃桃醒来后默认回到共同聊天，而不是停在生成结果页。
- “今天”从聊天上下文进入，完成后回到聊天，不成为独立任务首页。

### 10.2 小程序链路

- 微信卡片打开后能读取 inviteToken。
- 授权失败不丢邀请上下文。
- 授权后可展示同一个来源对象、桃桃预览、名字、性格和发起方。
- 接受、想改一下、不是这个三种结果都能写回 App。
- 小程序轻回应能以 `mini_return_card` 回到 App 聊天。
- 打开 App 失败时有复制链接和稍后继续路径。

### 10.3 状态与数据

- `TemporaryRoom` 可合并为正式 `ChatRoom`，且不丢 onboarding 进度。
- 已有账号、已有关系、重复邀请、邀请过期都有明确状态卡。
- App 重启、弱网、接口失败后能恢复到正确聊天和卡片状态。
- `AwakenSession`、`CoupleBond`、`TaotaoLife`、`MiniLinkSession`、`NestMemory` 状态一致。
- 另一半未确认前，不开放完整共同体验和共同记忆。

### 10.4 生命面板与记忆

- 桃桃页展示生命状态、来源、人格、动作、记忆成长和共同边界。
- 桃桃页不替代聊天，也不把邀请流程做成主内容。
- 聊天中可生成记忆草稿，且必须经过双方确认后写入共同记忆。
- 记忆确认后，生命面板和小窝有可见变化。
- 单人私密输入、高风险内容和未确认草稿不会自动沉淀为共同记忆。

### 10.5 视觉与交互

- 整体视觉符合优雅、克制、生活摄影感，不做传统情侣 App 贴纸化表达。
- 桃桃生命感来自动态、语气、停顿和真实生活场景，不靠花哨 AI 特效。
- 关键状态使用聊天卡片承载，不靠 toast。
- 每个状态只给一个主动作，次动作保持轻。
- 移动端 375、390、430 宽度下，聊天卡、登录卡、邀请卡、生命面板文字不溢出、不遮挡。

### 10.6 安全与边界

- 攻击、羞辱、监控、定位控制、自伤、暴力等请求触发降温或安全卡。
- 桃桃不站边、不裁决、不读取未授权聊天记录。
- 数据删除、解绑、来源照片删除有二次确认和影响说明。
- 付费或投喂不出现在争执、冷战、明显伤心等负面场景。
